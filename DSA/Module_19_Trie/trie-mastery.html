🧠 The Story — Google's Autocomplete Secret

Before we write a node class, understand why tries exist and what problem they solve better than hash maps.

Real World Story — The Autocomplete Engine

You type "co" in the search bar. In 50ms, Google shows you 10 suggestions.

Behind the scenes, there are billions of search queries stored. How do you find all queries starting with "co" in under 50ms?

Option A — Hash Map: Store each query as a key. To find all "co" prefixes, you'd need to scan every key and check if it starts with "co". That's billions of checks.

Option B — Sorted Array + Binary Search: Binary search to find "co", then scan forward while prefix matches. O(log n + k) where k = results. Not bad, but not great for modifications.

Option C — Trie: Every character is a node. Navigate "c" → "o" in 2 steps. Now you're at a node whose entire subtree contains every word starting with "co". Traverse the subtree to collect results. O(m + k) where m = prefix length. And it supports insert/delete in O(m) too.

Tries trade space for speed. They're the backbone of autocomplete, spell checkers, and IP routing tables.

The Key Insight — Shared Prefixes Share Nodes

Words "cat", "car", "cab" all share the path c→a. In a hash map, each is stored independently. In a trie, the c→a path is stored once and shared by all three words. This is the space-time tradeoff: slightly more space per node, but prefix operations become lightning-fast.

Each TrieNode stores: a dictionary of children (char → TrieNode) and a boolean marking if this node ends a word. That's it.

Python · Trie Implementation from Scratch
class TrieNode:
    def __init__(self):
        self.children = {}       # char → TrieNode
        self.is_end = False      # True if this node ends a complete word
        self.count = 0           # optional: words passing through here

class Trie:
    def __init__(self):
        self.root = TrieNode()

    def insert(self, word):
        node = self.root
        for char in word:
            if char not in node.children:
                node.children[char] = TrieNode()
            node = node.children[char]
            node.count += 1              # track prefix frequency
        node.is_end = True

    def search(self, word):
        node = self._navigate(word)
        return node is not None and node.is_end

    def starts_with(self, prefix):
        return self._navigate(prefix) is not None

    def _navigate(self, s):
        # Walk the trie following characters of s. Return final node or None.
        node = self.root
        for char in s:
            if char not in node.children: return None
            node = node.children[char]
        return node

    def autocomplete(self, prefix, limit=5):
        node = self._navigate(prefix)
        if not node: return []
        results = []
        def dfs(n, current):
            if len(results) == limit: return
            if n.is_end: results.append(current)
            for ch in sorted(n.children):       # sorted = alphabetical order
                dfs(n.children[ch], current + ch)
        dfs(node, prefix)
        return results

🧠 Intuition — why a Trie beats a hash set for prefixes

A hash set answers "is this exact word present?" in O(L), but it has no idea about prefixes — to find every word starting with "ca" it must scan the entire set. A trie turns the alphabet itself into the index: each level is one character position, and every node is a branch point shared by all words that agree up to that point. Walking "ca" takes 2 hops and lands you on the exact subtree containing cat, car, cab — nothing else is even visited.

🔵 Node = a shared prefix

The path root→c→a exists once. Inserting cat, car, cab only adds the diverging final letters. More words sharing a prefix ⇒ more reuse ⇒ better compression.

🟢 is_end = "a word stops here"

A node can be both a prefix and a complete word. is_end distinguishes "car" (a real word) from "ca" (just a prefix). Without it you can't tell the two apart.

🟠 When to reach for it

Prefix search, autocomplete, spell-check, IP routing, startsWith queries, or anything where many strings share leading characters.

🟣 The cost trade

Per-node dict overhead costs memory, but every operation is O(L) in the word length — totally independent of how many words you've stored. A million words don't slow a lookup down.

insert/search/startsWith = O(L) · L = length of the key, not the number of stored words
autocomplete(prefix) = O(P) to locate + O(matches) to collect · P = prefix length

🎬 Problem 1: Search Suggestions System (LeetCode 1268)

Given a list of products and a search word, return 3 suggestions after each character typed. Classic trie application.

💡
Two approaches compared:
Sort + Binary Search: Sort products. For each prefix, binary search for insertion point, then collect up to 3 lexicographically smallest matches. O(m log n) total.
Trie: Insert all products. For each prefix character, navigate trie and DFS for 3 suggestions. O(m²) insert time but O(m) per query after. Better if many queries.
products=["mobile","mouse","moneypot","monitor","mousepad"], word="mouse" After "m" → ["mobile","moneypot","monitor"] After "mo" → ["mobile","moneypot","monitor"] After "mou" → ["mouse","mousepad"] After "mous" → ["mouse","mousepad"] After "mouse"→ ["mouse","mousepad"]
Python · Search Suggestions — Trie Approach
def suggested_products(products, search_word):
    trie = Trie()
    for p in products:
        trie.insert(p)

    result = []
    for i in range(1, len(search_word) + 1):
        result.append(trie.autocomplete(search_word[:i], limit=3))
    return result

# Alternative: Sort + Binary Search (often preferred in interviews)
import bisect
def suggested_products_bs(products, search_word):
    products.sort()
    result, prefix = [], ''
    for c in search_word:
        prefix += c
        pos = bisect.bisect_left(products, prefix)
        suggestions = []
        for i in range(pos, min(pos + 3, len(products))):
            if products[i].startswith(prefix):
                suggestions.append(products[i])
            else: break
        result.append(suggestions)
    return result
🎬 Problem 2: Maximum XOR of Two Numbers (Trie of Bits)

A completely different use of Trie — store numbers as binary sequences to find maximum XOR in O(n × 32).

💡
The insight: XOR is maximized when bits differ. For each number, traverse a bit-trie from MSB to LSB, always trying to go to the opposite bit. If opposite bit exists, go there (XOR bit = 1). Otherwise, go same side (XOR bit = 0).
Numbers: [3,10,5,25,2,8] → Binary: [0011,1010,0101,11001,0010,1000] For number 5 (0101), trie navigation: Bit 4: want 1 (opposite of 0) → 1 exists → go right, xor bit=1 Bit 3: want 1 (opposite of 1) → 0 exists → go left, xor bit=1 (matched with 25=11001) ...result: 5 XOR 25 = 28, which is the maximum ✓
Python · Maximum XOR — Bit Trie
def find_maximum_xor(nums):
    # Build bit trie (each node has children[0] and children[1])
    root = {}
    for num in nums:
        node = root
        for i in range(31, -1, -1):     # MSB to LSB
            bit = (num >> i) & 1
            if bit not in node:
                node[bit] = {}
            node = node[bit]

    max_xor = 0
    for num in nums:
        node = root
        curr_xor = 0
        for i in range(31, -1, -1):
            bit = (num >> i) & 1
            opposite = 1 - bit
            if opposite in node:       # ideal: go opposite to maximize XOR
                curr_xor = (curr_xor << 1) | 1
                node = node[opposite]
            else:                       # forced: go same side
                curr_xor = (curr_xor << 1) | 0
                node = node[bit]
        max_xor = max(max_xor, curr_xor)
    return max_xor
⚡ Greedy bit-walk — maximize XOR by taking the OPPOSITE bit (query 5 = 0101)
opposite bit available → XOR bit = 1 forced same bit → XOR bit = 0 current bit position
Numbers are stored as fixed-width binary in a 0/1 trie. For each query we greedily steer toward the opposite bit at every level — each success adds a 1 to the running XOR, building the maximum pair. ▶ loops every 10s.

Design Autocomplete System (Hard)
Problem (LeetCode 642): Implement autocomplete with historical sentences and frequencies. inputCharacter(c) returns top-3 matching sentences in real-time. '#' ends input and saves the new sentence.
Trie with top-k at each node: Store sentences in a trie. Each node stores top-3 (sentence, frequency) pairs sorted by frequency. On each character typed, navigate the trie and return the current node's top-3. '#' means save the built sentence.

💻 Solution
Python · Autocomplete System
class AutocompleteSystem:
    def __init__(self, sentences, times):
        self.trie = {}
        self.freq = {}
        self.curr = ""
        for s, t in zip(sentences, times):
            self.freq[s] = t
            self._insert(s)

    def _insert(self, s):
        node = self.trie
        for c in s:
            if c not in node: node[c] = {}
            node = node[c]
        node['#'] = True          # end marker

    def _search_prefix(self, prefix):
        node = self.trie
        for c in prefix:
            if c not in node: return []
            node = node[c]
        # DFS to collect all sentences under this node
        results = []
        def dfs(n, path):
            if '#' in n: results.append(prefix[:-len(prefix)] + path)
            for ch, child in n.items():
                if ch != '#': dfs(child, path+ch)
        dfs(node, prefix)
        return sorted(results, key=lambda x: (-self.freq.get(x,0), x))[:3]

    def input(self, c):
        if c == '#':
            self.freq[self.curr] = self.freq.get(self.curr, 0) + 1
            self._insert(self.curr)
            self.curr = ""; return []
        self.curr += c
        return self._search_prefix(self.curr)
⌛ Complexity
Time: O(n × avg_len) build, O(total_chars) search Space: O(total_chars)

Word Search II — Find All Words in Grid
Problem (LeetCode 212): Given an m×n board and a list of words, find all words present in the board (adjacent cells, each cell at most once).
Trie + DFS with pruning: Insert all words into a trie. Start DFS from each cell. Navigate the trie simultaneously with the DFS — if the current character isn't in the trie at the current node, prune immediately. When we hit a word end marker, record the word. Prune discovered words (set is_end=False) to avoid duplicates.
🔍 Step-by-Step Trace

No → PRUNE Move to (0,1)='a': trie['o']['a']? Yes → Move to (1,1)='t': trie['o']['a']['t']? 💻 Solution
Python · Word Search II (Trie + DFS)
def find_words(board, words):
    trie = {}
    for w in words:
        node = trie
        for c in w: node = node.setdefault(c, {})
        node['$'] = w                      # store word at end node

    m, n = len(board), len(board[0])
    result = []
    def dfs(r, c, node):
        ch = board[r][c]
        if ch not in node: return       # prune
        node = node[ch]
        if '$' in node:
            result.append(node.pop('$'))   # found! remove to avoid dups
        board[r][c] = '#'               # mark visited
        for dr, dc in [(1,0),(-1,0),(0,1),(0,-1)]:
            nr, nc = r+dr, c+dc
            if 0 <= nr < m and 0 <= nc < n:
                dfs(nr, nc, node)
        board[r][c] = ch               # backtrack

    for r in range(m):
        for c in range(n): dfs(r, c, trie)
    return result
⌛ Complexity
Time: O(m×n × 4 × 3^(L-1)) Space: O(total word chars)
