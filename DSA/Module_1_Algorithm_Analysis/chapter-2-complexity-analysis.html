🧠 The Story — Measuring without a Stopwatch
Why Big O?

You can't benchmark on every possible computer with every possible input. Big O gives you a machine-independent, input-size-based measure of growth.

Two computers run your sort. One is 10× faster. But when n grows from 1,000 to 1,000,000, an O(n²) algorithm gets 1,000,000× slower while an O(n log n) algorithm gets only 20,000× slower. The algorithm matters far more than the hardware.

🔧 Rules for Calculating Complexity
Rule 1 — Drop constants: 3n + 7 = O(n), not O(3n+7) Rule 2 — Drop lower terms: n² + n = O(n²), not O(n²+n) Rule 3 — Sequential code adds: O(n) + O(n²) = O(n²) Rule 4 — Nested loops multiply: O(n) inside O(n) = O(n²) Rule 5 — Recursion: use recurrence relation T(n) = 2T(n/2) + O(n) → O(n log n) [Master Theorem, case 2] T(n) = T(n-1) + O(1) → O(n) [linear recursion] T(n) = 2T(n-1) + O(1) → O(2ⁿ) [exponential!] Master Theorem (for T(n) = a·T(n/b) + O(nᶜ)): If c < log_b(a) → O(n^log_b(a)) [recursion dominates] If c = log_b(a) → O(nᶜ log n) [equal work at every level] If c > log_b(a) → O(nᶜ) [combination step dominates]

Analyse Binary Search Complexity
Problem: Prove that binary search is O(log n). Why is the base of the log irrelevant in Big O notation?
Recurrence: Each step halves the search space. T(n) = T(n/2) + O(1). Solving: after k steps, space = n/2^k = 1, so k = log₂(n). For Big O, log₂n and log₁₀n differ by a constant factor log₂(10) ≈ 3.32 — constants are dropped.
🔍 Step-by-Step Trace
Binary search on n=1024 elements: Step 1: 1024 candidates Step 2: 512 candidates (halved) Step 3: 256 candidates Step 4: 128 candidates Step 5: 64 candidates Step 6: 32 candidates Step 7: 16 candidates Step 8: 8 candidates Step 9: 4 candidates Step 10: 2 candidates Step 11: 1 candidate → found or not found Total steps = log₂(1024) = 10 comparisons Recurrence: T(n) = T(n/2) + 1 Expand: 1 + 1 + 1 + ... = log₂(n) times → T(n) = O(log n) Base change: log₂n = log₁₀n / log₁₀2 = (constant) × log₁₀n → In Big O, all log bases are equivalent.
💻 Solution
Python · Binary search with step counter
def binary_search(arr, target):
    lo, hi, steps = 0, len(arr)-1, 0
    while lo <= hi:
        steps += 1
        mid = (lo + hi) // 2
        if   arr[mid] == target: print(f"Found in {steps} steps"); return mid
        elif arr[mid] <  target: lo = mid + 1
        else:                    hi = mid - 1
    return -1
# n=1024: ~10 steps.  n=1M: ~20 steps.  n=1B: ~30 steps.
⌛ Complexity
Time: O(log n) Space: O(1)

Amortized Analysis: Dynamic Array Append
Question: Python's list.append() is O(1) amortized but sometimes O(n). Explain amortized analysis and why the average is O(1).
Doubling strategy: When capacity is exceeded, double the array and copy all elements — O(n). But this happens so rarely that the cost, spread over all insertions, is O(1) per insertion. Total cost for n insertions = n + (copies at capacity) ≤ 2n → amortized O(1).
🔍 Step-by-Step Trace
Start with capacity=1, fill with n=8 elements: Insert 1: no resize. cost=1 Insert 2: resize (cap=1→2), copy 1. cost=1+1=2 Insert 3: resize (cap=2→4), copy 2. cost=1+2=3 Insert 4: no resize. cost=1 Insert 5: resize (cap=4→8), copy 4. cost=1+4=5 Insert 6: no resize. cost=1 Insert 7: no resize. cost=1 Insert 8: no resize. cost=1 Total cost = 1+2+3+1+5+1+1+1 = 15 for 8 inserts Average = 15/8 ≈ 1.875 ≈ O(1) amortized ✓ Copy costs = 1+2+4 = 7 (each is half the previous capacity) Sum of geometric series = 1+2+4+8+...+n ≤ 2n So total ≤ n (inserts) + 2n (copies) = 3n = O(n) total → O(1) amortized
💻 Solution
Python · Simulate dynamic array growth
class DynArray:
    def __init__(self):
        self.data, self.size, self.cap = [], 0, 1
    def append(self, x):
        if self.size == self.cap:        # resize!
            self.data = self.data + [None] * self.cap
            self.cap *= 2
        self.data[self.size] = x; self.size += 1

# Verify: count total operations for n appends
total = 0; cap = 1; size = 0
for i in range(1, 1025):
    if size == cap: total += size; cap *= 2  # copy cost
    total += 1; size += 1                              # insert cost
print(f"1024 inserts: {total} ops, avg={total/1024:.2f}")
# → 1024 inserts: 2046 ops, avg=1.999 ✓
⌛ Complexity
Time: O(1) amortized Space: O(n)

Space Complexity: Recursion Stack
Question: A recursive Fibonacci implementation fib(n) = fib(n-1) + fib(n-2) has O(2ⁿ) time complexity. What is its space complexity?
Stack frames, not total nodes. Space = maximum depth of recursion stack at any one time. The call tree has O(2ⁿ) nodes but the deepest path is n levels (always goes left before right). So space = O(n) stack frames.
🔍 Step-by-Step Trace
fib(5) call tree: fib(5) / \ fib(4) fib(3) / \ / \ fib(3) fib(2) fib(2) fib(1) / \ fib(2) fib(1) At any moment, only ONE root-to-leaf path is on the call stack. Max depth = n (the leftmost path fib(5)→fib(4)→fib(3)→fib(2)→fib(1)) Stack at deepest: [fib(5), fib(4), fib(3), fib(2), fib(1)] = 5 frames Space = O(n) — only the current active branch Time = O(2ⁿ) — total calls made Key insight: time counts ALL work ever done; space counts work IN PROGRESS simultaneously
💻 Solution
Python · Verify with sys.setrecursionlimit
import sys
sys.setrecursionlimit(1000)
depth_tracker = 0; max_depth = 0

def fib(n):
    global depth_tracker, max_depth
    depth_tracker += 1
    max_depth = max(max_depth, depth_tracker)
    if n <= 1: result = n
    else: result = fib(n-1) + fib(n-2)
    depth_tracker -= 1
    return result

fib(10)
print(f"Max stack depth for fib(10): {max_depth}")
# → Max stack depth for fib(10): 10 = O(n) ✓
⌛ Complexity
Time: O(2ⁿ) time Space: O(n) stack
