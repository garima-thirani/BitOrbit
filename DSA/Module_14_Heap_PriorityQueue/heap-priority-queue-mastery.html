🧠 The Story — The Hospital Emergency Room
Real World Story — Triage System

Patients arrive at the ER in random order. Each patient has a severity score. The most critical patient must always be treated first.

A regular queue is FIFO — the first to arrive is served first. But in the ER, a critical patient who just arrived must skip ahead of non-critical patients who've been waiting.

Heap = Priority Queue = ER Triage System.

A min-heap guarantees: the patient with the LOWEST severity number (most critical) is always at the front. Insert a new patient: O(log n) (they bubble up if critical enough). Treat next patient: O(log n) (remove root, restructure).

Python's heapq is a min-heap. For max-heap, negate values: push -val, pop and negate result.

Heap Property: Every Parent ≤ Children (Min-Heap)

The heap is stored as an array. For index i:
— Parent: (i-1)//2
— Left child: 2*i+1
— Right child: 2*i+2

Heap ≠ Sorted Array. The heap only guarantees the top element is min/max. Siblings have no ordering relationship. This makes insert/extract O(log n) instead of O(n).

💡 The Deep Intuition — Lazy Order Is Cheap Order

Here is the key bargain a heap strikes: keeping a collection fully sorted is expensive; keeping only the champion on top is cheap. Most of the time you don't need the whole ranking — you just need "give me the smallest, now give me the next smallest." A heap maintains the weakest possible promise that still answers that question: every parent beats its children. That single local rule, repeated everywhere, forces the global minimum to the very top — without ever sorting the rest. Think of it as a tournament bracket that you only patch along one root-to-leaf path whenever something changes.

🟡 When to reach for it

Any time the phrase is "top-K", "K-th largest/smallest", "merge K sorted things", "schedule by priority", "running median", or "repeatedly take the best option" (Dijkstra, Prim, Huffman). If you keep wanting min() or max() on a changing collection, that's a heap.

🔵 The array trick

A heap is a complete binary tree, so it needs no pointers — it lives in a flat array. For index i: parent is (i-1)//2, children are 2i+1 and 2i+2. "Complete" means no gaps, which keeps height at exactly ⌊log₂n⌋.

🟣 Two repair moves

sift-up (after insert): a new value lands at the end and swaps with its parent while it's smaller, climbing one path up. sift-down (after extract): the last value fills the empty root and sinks by swapping with its smaller child.

peek min  = O(1)
insert    = O(log n)  sift-up
extract min = O(log n)  sift-down
heapify   = O(n) bottom-up

🟢 The counter-intuitive part

To track the K largest, you keep a min-heap of size K (not a max-heap!). Its top is the weakest of your current champions; when a bigger value arrives, you evict that weakest. After one pass the heap holds exactly the top K, and its root is the K-th largest.

🌍 Heaps quietly running real systems
  • Hospital ER triage: the most critical patient is treated next regardless of arrival time — a priority queue keyed on severity.
  • Your OS task scheduler: the CPU repeatedly picks the highest-priority ready thread from a heap, thousands of times a second.
  • Google Maps / GPS routing: Dijkstra's algorithm pulls the closest-unvisited node from a min-heap on every step.
  • Event-driven simulations & game engines: a heap keyed on timestamp always returns the next event to process.
  • "Trending now" & Top-10 dashboards: a size-K min-heap streams over millions of items keeping only the K hottest — no full sort needed.
  • File compression (Huffman coding): repeatedly merge the two least-frequent symbols, pulled from a min-heap.
⚠️ Easy ways to get burned

1. No built-in max-heap in Python. heapq is min-only; push -value (or a wrapper) to fake a max-heap, and remember to negate on the way out. 2. Comparing un-comparable payloads. When you push tuples like (priority, item) and two priorities tie, Python compares the items next — which explodes if they're objects. Insert a unique tie-breaker: (priority, count, item). 3. A heap is not sorted. Iterating the underlying array does not give sorted order — only repeated heappop does.

Python · heapq — Complete API
import heapq

# Min-heap operations (Python default)
h = []
heapq.heappush(h, 5)           # insert: O(log n)
heapq.heappush(h, 2)
heapq.heappush(h, 8)
top = h[0]                     # peek min: O(1)  → 2
val = heapq.heappop(h)         # extract min: O(log n) → 2

# Build heap from list: O(n) — faster than n inserts!
arr = [5, 2, 8, 1, 9, 3]
heapq.heapify(arr)             # transforms arr in-place to min-heap

# Max-heap: negate values
max_heap = []
heapq.heappush(max_heap, -5)   # push -val
heapq.heappush(max_heap, -2)
max_val = -heapq.heappop(max_heap)  # negate on pop → 5

# K largest elements: use min-heap of size k
def k_largest(nums, k):
    return heapq.nlargest(k, nums)   # built-in: O(n log k)

# Custom objects: push tuples — first element is the key
tasks = [(3, "low"), (1, "critical"), (2, "medium")]
heapq.heapify(tasks)
priority, name = heapq.heappop(tasks)  # → (1, "critical")
🎬 Insert 1 — sift UP: a newcomer climbs one path to its level
Only one path moves. The newcomer touches just its ancestors — at most ⌊log₂n⌋ swaps — and every other element stays put. That locality is why insertion is O(log n), not O(n). ▶ loops every 10s.
🎬 extract_min — sift DOWN: the last element fills the hole and sinks
Extract is sift-up in reverse. Take the root (the answer), drop the last leaf into the empty root, then let it sink by always swapping with its smaller child until both children are larger. Again, just one root-to-leaf path is touched — O(log n). ▶ loops every 9s.

🎬 Problem 1: Kth Largest Element
💡
Counter-intuitive trick: To find Kth LARGEST, use a min-heap of size K. Why? The min of the top-K largest elements is the Kth largest. Maintain heap size K: if new element > heap min, pop min and push new element.
nums=[3,2,1,5,6,4], k=2. Find 2nd largest. min-heap of size 2: push 3 → [3] push 2 → [2,3] (size=2) push 1: 1<heap_min(2) → skip push 5: 5>heap_min(2) → pop 2, push 5 → [3,5] push 6: 6>3 → pop 3, push 6 → [5,6] push 4: 4<5 → skip Heap top = 5 → 2nd largest ✓
Python · Kth Largest — O(n log k)
def find_kth_largest(nums, k):
    min_heap = nums[:k]
    heapq.heapify(min_heap)              # build initial heap of k elements: O(k)

    for num in nums[k:]:
        if num > min_heap[0]:            # larger than current kth: replace
            heapq.heapreplace(min_heap, num)  # pop min + push in one O(log k)

    return min_heap[0]                   # top of heap = kth largest
🎬 Problem 2: Merge K Sorted Lists
Python · Merge K Sorted Lists — O(N log K)
def merge_k_lists(lists):
    heap = []
    dummy = ListNode(0)
    cur = dummy

    # Initialize heap with first node of each list
    for i, node in enumerate(lists):
        if node:
            heapq.heappush(heap, (node.val, i, node))  # (value, list_idx, node)

    while heap:
        val, i, node = heapq.heappop(heap)    # always get global minimum
        cur.next = node
        cur = cur.next
        if node.next:
            heapq.heappush(heap, (node.next.val, i, node.next))  # add next from same list

    return dummy.next

# Why O(N log K): N total nodes, each pushed/popped once from K-element heap.
# The i index in tuple breaks ties between equal values (avoids comparing ListNode).
🎬 Problem 3: Find Median from Data Stream
📊
Brilliant two-heap trick: — max_heap: stores the LOWER half (negate values for max) — min_heap: stores the UPPER half — Invariant: all elements in lower half ≤ all elements in upper half — Balance: sizes differ by at most 1 — Median: if equal sizes → average of both tops; else top of larger heap.
Stream: 1, 2, 3, 4, 5 After 1: lower=[-1], upper=[] → median=1 After 2: lower=[-1], upper=[2] → median=1.5 After 3: lower=[-2,-1], upper=[3] → median=2 (lower top) After 4: lower=[-2,-1], upper=[3,4] → median=2.5 After 5: lower=[-3,-2,-1], upper=[4,5] → median=3
Python · Median Finder — Two Heaps
class MedianFinder:
    def __init__(self):
        self.lo = []   # max-heap (negated): lower half
        self.hi = []   # min-heap:           upper half

    def addNum(self, num):
        heapq.heappush(self.lo, -num)           # push to lower half
        heapq.heappush(self.hi, -heapq.heappop(self.lo))  # balance: move max of lower to upper
        if len(self.hi) > len(self.lo):         # keep lower >= upper in size
            heapq.heappush(self.lo, -heapq.heappop(self.hi))

    def findMedian(self):
        if len(self.lo) > len(self.hi):
            return -self.lo[0]                  # odd: lower has extra
        return (-self.lo[0] + self.hi[0]) / 2  # even: average of both tops
🎬 Two heaps guarding a wall — the median lives where they meet
The median is always touching the wall. Keep the smaller half in a max-heap and the larger half in a min-heap, balanced so their sizes differ by ≤ 1. Then the answer is O(1): if one side is bigger, its top is the median; if tied, average the two tops. Each insert is one push plus a rebalance — O(log n). ▶ loops every 13s.

K Closest Points to Origin
Problem (LeetCode 973): Given a list of points, return the k closest to the origin (0,0). Distance = sqrt(x²+y²). You don't need to sort — just return ANY k closest.
Max-heap of size k: Maintain a max-heap of the k smallest distances seen so far. For each new point: if it's closer than the heap's max (farthest in current k), evict max and insert new. Python's heapq is a min-heap, so negate distances. Final heap holds k closest.
🔍 Step-by-Step Trace
points=[(1,3),(-2,2),(5,8),(0,1)] k=2 Distances^2: (1,3)→10, (-2,2)→8, (5,8)→89, (0,1)→1 Process with max-heap (using negated sq dist): (1,3) d²=10: heap=[(-10,(1,3))] (-2,2) d²=8: heap=[(-10,(1,3)),(-8,(-2,2))] size=k (5,8) d²=89: 89>10 (current max) → don't insert (0,1) d²=1: 1<10 → pop (-10,(1,3)), push (-1,(0,1)) heap=[(-8,(-2,2)),(-1,(0,1))] Extract points from heap: [(-2,2),(0,1)] ✓
💻 Solution
Python · K Closest Points (Max-Heap)
import heapq
def k_closest(points, k):
    max_heap = []
    for x, y in points:
        dist = x*x + y*y              # no sqrt needed (monotone)
        heapq.heappush(max_heap, (-dist, x, y))    # negate for max-heap
        if len(max_heap) > k:
            heapq.heappop(max_heap)   # evict farthest point
    return [[x, y] for _, x, y in max_heap]
⌛ Complexity
Time: O(n log k) Space: O(k)
2
Task Scheduler — Minimum Time
Problem (LeetCode 621): CPU executes tasks, same task must wait n intervals before re-execution (cooldown). Given list of tasks, find minimum time to finish all. E.g. ['A','A','A','B','B','B'] n=2 → 8.
Math formula + Greedy insight: The task with highest frequency is the bottleneck. It creates 'frames' of (n+1) each. Fill frames with other tasks (or idle). Formula: max((max_freq-1)*(n+1)+count_of_max, len(tasks)). Heap approach: always pick highest-frequency available task.
🔍 Step-by-Step Trace
tasks=['A','A','A','B','B','B'] n=2 freq: A=3, B=3, max_freq=3 Frames of (n+1)=3: Frame 1: A, B, _ Frame 2: A, B, _ Frame 3: A, B Formula: max((3-1)*(2+1)+2, 6) = max(8, 6) = 8 ✓ Timeline: A→B→idle→A→B→idle→A→B (8 units) Greedy heap simulation: heap: [(-3,A),(-3,B)] time=0 cooldown_q=[] tick 1: pop (-3,A)→A, schedule A again at t=3. time=1 tick 2: pop (-3,B)→B, schedule B at t=4. time=2 tick 3: heap empty, cooldown has (A,3) not ready → idle. time=3 release A from cooldown → push (-2,A) tick 4: pop (-2,A)→A... → time=8 ✓
💻 Solution
Python · Task Scheduler (Formula)
from collections import Counter
def least_interval(tasks, n):
    counts = Counter(tasks)
    max_freq = max(counts.values())
    # How many tasks share the maximum frequency?
    count_max = sum(1 for v in counts.values() if v == max_freq)
    # Frames: (max_freq-1) full frames + partial last frame
    formula = (max_freq - 1) * (n + 1) + count_max
    return max(formula, len(tasks))  # can't do less than total tasks
⌛ Complexity
Time: O(n) Space: O(1)

Sliding Window Maximum (Heap approach)
Problem (LeetCode 239): Max in each window of size k over array. Heap alternative: max-heap with lazy deletion — mark stale elements and skip them when they appear at top.
Max-Heap with lazy deletion: Push (value, index). To find window max: peek at top; if its index is outside window, pop it (lazy deletion). O(n log n) vs deque's O(n) but heap is more intuitive. The deque is optimal but heap is acceptable if you can't recall deque.
🔍 Step-by-Step Trace
nums=[1,3,-1,-3,5,3,6,7] k=3 Max-heap (negate values for Python's min-heap): i=0: push (-1,0). heap=[(-1,0)] i=1: push (-3,1). heap=[(-3,1),(-1,0)] i=2: push (1,2). heap=[(-3,1),(-1,0),(1,2)] Window [0,2]: peek=(-3,1), idx=1 in [0,2] → max=3 ✓ i=3: push (3,3). heap=[(-3,1),(-3,3),(-1,0),(1,2)] Window [1,3]: peek=(-3,1), idx=1 in [1,3] → max=3 ✓ i=4: push (-5,4). heap=[(-5,4),...] Window [2,4]: peek=(-5,4), idx=4 in [2,4] → max=5 ✓ (next peek would be (-3,1), idx=1 NOT in [2,4] → pop, lazy delete)
💻 Solution
Python · Sliding Window Max (Max-Heap, lazy delete)
import heapq
def max_sliding_window_heap(nums, k):
    heap = []                          # max-heap via negated values
    result = []
    for i, n in enumerate(nums):
        heapq.heappush(heap, (-n, i))
        if i >= k - 1:
            # Lazy-delete entries outside window
            while heap[0][1] < i - k + 1:
                heapq.heappop(heap)
            result.append(-heap[0][0])  # top = current window max
    return result
⌛ Complexity
Time: O(n log n) Space: O(n)
