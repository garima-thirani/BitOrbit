📖 The Story — Choosing a Backpack
Trade-off Thinking

Light. Cheap. Durable. Pick two.

Suppose you need a backpack for a Himalayan trek. You want it to be extremely light, extremely cheap, and extremely durable. Unfortunately, you rarely get all three.

If it is light and durable, it is expensive. If it is cheap and durable, it is heavy. If it is cheap and light, it may not last.

Software architecture works exactly the same way. Every engineering decision optimizes one thing while sacrificing another. Junior engineers ask "Which database should I use?" Senior engineers ask "What problem am I optimizing for?"

🧠 The CAP Theorem
CAP stands for: Consistency · Availability · Partition Tolerance
The theorem states: In a distributed system, you can guarantee at most two of these three properties simultaneously.
CAP Theorem: In a distributed system with network partitions (which always happen), you choose between Consistency and Availability.

⏸
CP System (Consistency + Partition Tolerance)
The network between two data centers is broken. When a user writes to Region A, Region B cannot verify the data. A CP system rejects the request until the partition heals — because serving stale data is worse than serving no data. Example: Banking, where ₹10,000 must not disappear.

💡 Modern systems assume partitions WILL happen. The real choice is between Consistency and Availability during failures.

⚡ Latency vs Consistency
Real-World Trade-off

Should Instagram likes update instantly?

Consider an online banking system. A user transfers ₹10,000. Would you rather return the result instantly but risk showing stale data? Or wait slightly longer to guarantee the correct balance?

For banking, correctness is more important.

Now think about Instagram likes. If your like count updates one second later, most users won't notice. Here, lower latency matters more than immediate consistency.

The right trade-off depends entirely on the use case.

🏛 Bitly High-Level Design
Bitly is a URL shortening service. It takes a long URL and returns a short, unique alias. When users visit the short URL, they are redirected to the original URL. It handles billions of redirects daily.
📋 Requirements

✅ Functional

• Shorten long URLs
• Redirect short URLs
• Basic analytics (click count)
• Custom aliases (optional)

📊 Non-Functional

• Very low redirect latency (<50ms)
• High availability (99.99%)
• Billions of redirects
• Unique short IDs (collision-free)

🏗 High-Level Architecture
Bitly HLD: Load balancer distributes traffic. URL creation generates IDs and stores mappings. Redirects served from Redis cache. Analytics processed asynchronously.
🔍 Why Each Component Exists
  • Load Balancer — Distributes incoming traffic across multiple API servers. Prevents any single server from being overwhelmed.
  • URL Creation Service — Validates the long URL, calls the ID generator, and stores the mapping in PostgreSQL. Returns the short URL to the client.
  • ID Generator — Ensures uniqueness. Strategies include Base62 encoding (encode database auto-increment ID), Snowflake IDs (distributed unique IDs), or random IDs with collision detection.
  • Redis Cache — Stores frequently accessed URL mappings. Most redirects (90%+) become cache hits, eliminating database load. TTL-based eviction handles stale entries.
  • PostgreSQL — Persistent storage for all URL mappings. Provides durability and serves as the source of truth for cache misses.
  • Analytics Queue — Redirects must remain fast. Click analytics (timestamps, referrers, geolocation) are processed asynchronously via a message queue.
📋 Request Flow

URL Creation Flow

1
Client sends POST /v1/urls with { "longUrl": "https://..." }
2
API server validates the URL format
3
ID Generator creates a unique short ID (Base62 or Snowflake)
4
Mapping stored in PostgreSQL: short_id → long_url
5
Short URL returned to client: https://sho.rt/ab12CD

Redirect Flow

1
Client sends GET /ab12CD
2
Redirect Service checks Redis Cache first
3
Cache Hit (90%+) → Immediate redirect (HTTP 302)
4
Cache Miss → Query PostgreSQL → Update Cache → Redirect
5
Analytics event pushed to queue (async, non-blocking)
💡 Why This Architecture Scales
📖 Reads dominate writes
5:1 read-to-write ratio. Caching dramatically reduces database load.
⚡ Redis absorbs hot traffic
90%+ cache hit rate means the database only handles 10% of read traffic.
🔧 Stateless API servers
API servers can scale horizontally. Just add more behind the load balancer.
📤 Async analytics
Analytics is processed in the background. Redirect latency is never affected.
🧩 Independent scaling
Each component scales independently based on its own load patterns.
🎯 Interview Walkthrough — The Right Approach
FAANG Interview Strategy

The 10-Step System Design Flow

1. Clarify requirements — Ask questions before drawing.
2. Estimate traffic — DAU, QPS, storage, bandwidth.
3. Define APIs — REST contracts for creation and redirect.
4. Draw HLD — Load balancer, services, cache, database.
5. Explain storage — Why PostgreSQL? Schema design.
6. Add caching — Redis for read-heavy redirects.
7. Identify bottlenecks — Database under high traffic.
8. Discuss scaling — Read replicas, sharding if needed.
9. Mention monitoring — Latency tracking, cache hit rates.
10. Explain trade-offs — Why this design? What did we sacrifice?

Key insight: Technology names are introduced only after the reasoning. You don't say "Redis" — you say "We need a fast cache for read-heavy redirects, and Redis with TTL is appropriate here."

🚨 Common Mistakes
❌ Saying "Microservices" Without Justification
Microservices add complexity. Justify each service boundary.
❌ Ignoring Failure Scenarios
What happens when Redis goes down? Cache stampede?
❌ Forgetting Cache Invalidation
TTL-based eviction is simple. How do you handle updates?
❌ Designing for Billions Without Estimates
Show your math. Numbers justify architecture.
❌ Treating CAP as "Pick Any Two"
Partitions will happen. The real choice is CP vs AP.
