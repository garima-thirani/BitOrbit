Session Affinity

Returning to the Same Banker

Imagine you visit a bank every week. Only one employee knows your paperwork, preferences, and history. Whenever you arrive, the receptionist always sends you to the same employee. Initially, this seems convenient. Now imagine that employee is sick. Nobody else knows your case. Your work stops. This is exactly how sticky sessions behave.

Sticky Session: Once a client is assigned to a server, future requests from that client continue going to the same server. The server becomes responsible for maintaining that user's session.
⚠ Why Were Sticky Sessions Introduced?

Early web applications stored login sessions in server memory. If the next request reached a different server, the session didn't exist — the user appeared logged out. Sticky sessions ensured every request returned to the same machine.

❌ Problems with Sticky Sessions
📊 Poor Load Distribution
One server may accumulate many active users while others remain underutilized.
💀 Single Point of User State
If the assigned server crashes, the user's session is lost.
📈 Difficult Auto Scaling
Adding new servers doesn't help existing users tied to old servers.
🚀 Complicated Deployments
Rolling updates become harder because active sessions cannot easily move between servers.
✅ Modern Alternatives

🔴 Redis Session Store

Move session data out of the application into a shared Redis cluster. Any server can retrieve the user's session.

🔵 JWT (JSON Web Token)

Encode user identity in a signed token. The client sends it with every request. No server-side session memory needed. Excellent horizontal scalability.

🎯 Interview Callout
Q
"Would you recommend sticky sessions for a new microservice architecture?"
"No. I would design stateless services and externalize session state using Redis or JWTs. Sticky sessions increase coupling and reduce flexibility."
🧠 Memory Trick
Sticky Sessions = Sticky Problems. Useful for legacy systems. Avoid for modern distributed architectures.
