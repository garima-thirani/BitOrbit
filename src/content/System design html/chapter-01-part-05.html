📖 The Story — Ordering Food at a Restaurant
API as a Contract

The menu is the contract between customer and kitchen.

Imagine walking into a restaurant. You don't walk into the kitchen and tell the chef how to cook. Instead, you interact with the menu. The menu is a contract. It tells you what operations are available, what inputs are expected, and what outputs you will receive.

The kitchen can completely change how food is prepared without changing the menu. APIs work exactly the same way. The client only knows the contract — not the implementation. This is the foundation of loose coupling.

🧠 What Is an API Contract?
Formal Definition: An API is a well-defined interface that allows one software component to communicate with another through standardized requests and responses. The important word is interface. Interfaces reduce coupling.

An API contract specifies:

📍 Endpoint
The URL path where the resource lives
🔧 HTTP Method
GET, POST, PUT, PATCH, or DELETE
📥 Request Parameters
Headers, query params, body schema
🔐 Authentication
How the client proves identity
✅ Validation Rules
Required fields, data types, constraints
📤 Response Schema
Shape of success responses
⚠ Error Codes
Status codes and error messages
🔌 REST API Contract Example
POST
/v1/urls
Create a shortened URL from a long URL
Request Body
{ "longUrl": "https://example.com/blog/system-design" }
Success Response (201)
{ "id": "ab12CD", "shortUrl": "https://sho.rt/ab12CD", "createdAt": "2026-01-07T12:00:00Z" }
Error Status Codes
400 Bad Request 401 Unauthorized 409 Duplicate 429 Rate Limited 500 Internal Error
GET
/v1/urls/{id}
Retrieve the original URL for a short ID
Success Response (200)
{ "id": "ab12CD", "longUrl": "https://example.com/blog/system-design", "shortUrl": "https://sho.rt/ab12CD", "clicks": 1542 }

⚡ HTTP Methods Reference
GET
Read a resource. Idempotent. Safe.
POST
Create a new resource. Not idempotent.
PUT
Replace a resource. Idempotent.
PATCH
Partial update. May not be idempotent.
DELETE
Remove a resource. Idempotent.
🏛 Client-Server Architecture
Client requests → Server processes → Database persists → Response returned to client
📋 Responsibilities

📱 Client

• UI rendering & user interaction
• Basic input validation
• Managing local state
• Handling network requests
• Caching responses locally

🖥 Server

• Authentication & authorization
• Business logic & rules
• Data persistence
• Security enforcement
• Auditing & logging

⚠ Golden Rule

Never trust the client. Always validate on the server. Client-side validation is for user experience. Server-side validation is for correctness and security.

🎯 Principles of Good API Design
Remember CRAFT: Consistent naming · Resource oriented · Authentication · Failure handling · Typed responses
1
Resource-Oriented URLs
Good: GET /users/42 · Bad: GET /getUser?id=42
2
Use Correct HTTP Methods
GET for read, POST for create, PUT for full update, PATCH for partial, DELETE for removal
3
Idempotency
Calling the same request multiple times should not create inconsistent state. PUT is idempotent, POST usually isn't. Use idempotency keys for payments.
4
Versioning
Never break existing clients. Use URL versioning: /v1/users, /v2/users
🏭 Production Example — Amazon Checkout
Real World

APIs Define Service Boundaries

During checkout at Amazon, multiple services interact through API contracts:

1. Client calls Payment API → validates request
2. Inventory Service → reserves stock
3. Order Service → creates the order
4. Notification Service → sends confirmation

Each interaction is governed by API contracts. Without stable contracts, hundreds of teams cannot develop independently.

💡 Why API Design Comes Before Architecture
Services are discovered through responsibilities exposed by APIs. Contracts define service boundaries.
🚨 Common Mistakes
❌ Designing APIs Around Database Tables
APIs should reflect business logic, not database schema.
❌ Ignoring Error Responses
Every API must document what errors clients can expect.
❌ Using POST for Everything
Use the correct HTTP method for each operation.
❌ Forgetting Authentication
Never expose endpoints without proper auth checks.
❌ Breaking Backward Compatibility
Version your APIs before making breaking changes.
🧪 Test Your Knowledge
1. Which HTTP method is naturally idempotent?
A) POST
B) PUT — replacing the same resource multiple times results in the same state
C) PATCH
D) All methods are idempotent
PUT is idempotent — calling PUT /users/42 with the same body always produces the same state. POST is NOT idempotent — it creates a new resource each time.
2. What is the best URL design for fetching a user with ID 42?
A) /getUser?id=42
B) /v1/users/42 — resource-oriented with versioning
C) /fetchUser/42
D) /userData?userId=42
Resource-oriented URLs reflect the entity (users) and use the ID as a path parameter. Versioning (/v1/) ensures backward compatibility.
3. True or False: Client-side validation is sufficient for security.
A) True — clients know the data best
B) False — always validate on the server; never trust the client
C) True — with HTTPS encryption
D) True — if using TypeScript
Never trust the client. Client validation is for UX. Server validation is for correctness and security.
