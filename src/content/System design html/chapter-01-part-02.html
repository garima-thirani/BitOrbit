📖 The Story — The Whiteboard That Should Never Have Been Drawn
The Requirements Mindset

Architecture is discovered, not invented.

Imagine walking into your first day as the lead architect at a startup. The CEO smiles and says, "We're building the next Uber." Immediately, one engineer rushes toward the whiteboard. Within minutes the board is filled with boxes: Redis, Kafka, Kubernetes, PostgreSQL, API Gateway, Load Balancer, CDN. Everything looks impressive. Except one thing. Nobody knows what they're building.

A second engineer quietly watches. Instead of opening a diagramming tool, they open a notebook. The first sentence they write isn't an architecture. It's a question: "Who are our users?" Then another: "Which countries are we launching in?" Then another: "How many daily active users should we expect?"

Within ten minutes the room changes. The conversation shifts away from technology and toward the business. Only now does architecture begin to emerge. That is the difference between drawing systems and designing them.

🧭 The Architect's Mental Model
A business idea is never translated directly into architecture. It first becomes questions, then requirements, then constraints, and only finally becomes software architecture.
🎯 Functional vs Non-Functional Requirements

✅ Functional Requirements

WHAT the system must do — user-visible behavior and business capabilities.

• Users register and login
• Drivers accept rides
• URLs are shortened
• Videos can be uploaded
• Users receive notifications

📊 Non-Functional Requirements

HOW WELL the system should perform — quality attributes.

• Scalability — handles 10× growth
• Availability — 99.99% uptime
• Latency — <100ms response
• Durability — no data loss
• Security — encrypted at rest

Key Insight: A URL shortener for 1,000 employees and Bitly serving millions have identical features but radically different non-functional requirements. The features don't change — the performance goals do.
📋 Interview Clarification Checklist

Always begin with these 10 questions before drawing any architecture:

👥 Who are the users?
Customers, admins, third-party systems?
🎯 Core use cases?
What are the primary features?
📊 Daily active users?
DAU determines scale requirements
⚡ Peak QPS?
Design for peak, not average
📖 Read-heavy or write-heavy?
Drives caching and replication strategy
⏱ Latency target?
P99 response time requirement
🛡 Availability target?
99.9% vs 99.99% changes everything
💾 Data retention?
How long must data be stored?
💰 Budget constraints?
Startup vs enterprise budget
🔒 Compliance?
GDPR, HIPAA, SOC2 requirements
🧠 The PEDALS Framework

A simple way to remember common non-functional concerns:

P — Performance
Response time, throughput, efficiency
E — Extensibility
Can new features be added easily?
D — Durability
Is data safe from loss or corruption?
A — Availability
What percentage of uptime is required?
L — Latency
How fast should responses be?
S — Security / Scalability
Is data protected? Can it grow?
📋 The RESHADED Framework

A comprehensive interview checklist to avoid missing architectural constraints:

R — Reliability
System works correctly under failure
E — Efficiency
Optimal resource utilization
S — Scalability
Handles growth in users/data
H — High Availability
Minimal downtime
A — Auditability
Track who did what and when
D — Durability
No data loss on failure
E — Extensibility
Easy to add/modify features
D — Disaster Recovery
Recover from catastrophic failures
🌍 Production Insight
Real World

Amazon's Design Reviews Begin with Assumptions

Amazon's design reviews frequently spend more time validating assumptions than discussing technology. Incorrect assumptions lead to expensive redesigns; correct requirements naturally narrow the architectural choices.

Key insight: A five-minute conversation can prevent months of expensive redesign. The best architects ask more questions than they answer.

🚨 Common Mistakes
❌ Starting with Technology
Never begin with Redis, Kafka, or Kubernetes. Begin with the problem.
❌ Ignoring Non-Functional Requirements
Features alone don't determine architecture. Scale, latency, and availability do.
❌ Not Asking Enough Questions
Every unanswered question is a risk in the architecture.
❌ Confusing Business Goals with Requirements
"Build Uber" is a goal. "Book a ride" is a requirement.
🧪 Test Your Knowledge
1. Which of the following is a non-functional requirement?
A) Users can upload profile photos
B) Drivers can accept ride requests
C) The system must respond within 200ms at P99
D) Users can reset their password
Non-functional requirements describe HOW WELL the system performs. Response time is a quality attribute, not a feature.
2. What does the "A" in PEDALS stand for?
A) Authentication
B) Availability — the percentage of uptime required
C) Algorithms
D) Automation
PEDALS = Performance, Extensibility, Durability, Availability, Latency, Security/Scalability
3. True or False: A URL shortener for 100 employees has the same architecture as Bitly serving 100M users.
A) True — the features are identical
B) False — non-functional requirements (scale, latency, availability) radically change the architecture
C) True — both need a database
D) False — only the programming language changes
Features may be identical, but the non-functional requirements (millions of QPS, 99.99% availability, global distribution) demand a completely different architecture.
