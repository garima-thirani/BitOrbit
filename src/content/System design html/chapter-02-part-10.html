📋 Chapter 2 Summary
🧠 Scalability
The ability to maintain performance as demand increases. Scale follows success.
⬆ Vertical Scaling
Bigger Box — upgrade CPU, RAM, disk. Simple but limited by physical constraints.
➡ Horizontal Scaling
More Boxes — add servers behind a load balancer. Virtually unlimited but adds complexity.
🔌 Stateless Architecture
State belongs to the data layer, not the application layer. Essential for horizontal scaling.
⚖ Load Balancers
Traffic police — distribute requests, perform health checks, enable failover.
🔒 Sticky Sessions
Legacy approach. Modern systems use Redis or JWT for session management.
📈 Auto Scaling
Measure → Decide → Scale → Stabilize. Reactive and predictive approaches.
🌍 Geographic Distribution
Deploy globally for lower latency and disaster recovery. Active-Active vs Active-Passive.
🧪 Final Quiz
1. Which scaling approach would you recommend for a startup with 100 users?
A) Kubernetes with 10 nodes
B) Vertical scaling — start with a single powerful server
C) Multi-region deployment
D) Microservices from day one
Start simple. Add horizontal scaling only when you hit the limits of vertical scaling.
2. What is the key requirement for horizontal scaling of application servers?
A) A fast database
B) Stateless services
C) More RAM
D) SSD storage
Stateless services allow any server to handle any request, enabling horizontal scaling, load balancing, and auto scaling.
