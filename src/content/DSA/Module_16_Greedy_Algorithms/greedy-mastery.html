🧠 The Story — The Greedy Gamble

Before writing any greedy algorithm, ask yourself one question: "Does grabbing the best thing now guarantee the best outcome overall?"

Real World Story — The Movie Director

You have a single filming studio. 8 movies want to use it today.

Each movie has a start time and end time. You want to film as many movies as possible (maximize utilization).

Greedy attempt 1 — Start earliest: Pick the movie that starts first. Sounds reasonable. But what if the earliest-starting movie runs all day? You'd only film 1 movie.

Greedy attempt 2 — Shortest first: Pick the shortest movie. Also reasonable. But a short movie might start right in the middle of the day, blocking two longer early-morning slots.

Greedy attempt 3 — Finish earliest: Always pick the movie that finishes soonest. This is optimal! Why? Because a movie that ends early leaves the most future time available. You are maximizing future flexibility.

This is the Activity Selection Problem — the archetypal greedy algorithm — and the proof of correctness is the template for proving every greedy algorithm.

The Greedy Correctness Test — Two Properties Needed
  • Greedy Choice Property: The locally optimal choice at each step can be part of a globally optimal solution. (Proof: Exchange argument — assume OPT differs from greedy, show swapping doesn't worsen it.)
  • Optimal Substructure: After making the greedy choice, the remaining subproblem also has optimal substructure. The same logic applies recursively.

If either property fails, greedy gives wrong answers. This is why coin change (US denominations work, arbitrary ones don't) and 0/1 Knapsack require DP instead.

🧠 Intuition — the exchange argument, the heart of every greedy proof

Greedy feels like cheating: you make the best-looking choice right now and never reconsider. Why would never looking ahead ever be optimal? The answer is the exchange argument. Suppose some optimal solution OPT disagrees with your first greedy pick. Take OPT and swap your greedy choice in for whatever OPT did. If you can show this swap never makes OPT worse, then a solution just as good as OPT exists that agrees with greedy on the first step. Repeat down the line and greedy matches OPT everywhere — so greedy is optimal. The whole skill is choosing the right "best" metric so that swap always holds: earliest finish time for scheduling, highest value-per-weight for fractional knapsack, furthest reach for jump game.

🔵 Greedy choice property

A locally optimal pick can be extended to a global optimum. Proven by the exchange/swap argument — never by "it seems reasonable".

🟢 Optimal substructure

After committing the greedy choice, what remains is a smaller instance of the same problem. The same rule applies recursively.

🟠 Pick the right metric

The art is the sort key. Earliest finish, max ratio, furthest reach, smallest deadline — the metric must make the swap argument go through.

🟣 When greedy lies

0/1 Knapsack and arbitrary-denomination coin change break the greedy choice property — a locally best grab blocks a better whole. Use DP there.

greedy works ⟺ greedy-choice property + optimal substructure
proof template: assume OPT ≠ greedy at step 1 → swap greedy in → OPT not worse → induct
⚠ The classic trap

"It passed my examples" is not a proof. Greedy that looks right can be subtly wrong (coin change with coins [1,3,4] for amount 6: greedy 4+1+1=3 coins, optimal 3+3=2 coins). Always justify with an exchange argument or fall back to DP.

🎬 Problem 1: Activity Selection — Maximum Non-Overlapping Events

The foundation. Master this and you understand why "end-time first" is the right greedy choice.

💡
Why sort by end time (not start time, not duration)?
Think of it this way: when you pick an activity, you're "consuming" studio time up to its end time. Choosing the activity with the earliest end time leaves the maximum remaining time for future activities. Any other choice can only leave less time. This is the exchange argument: if an optimal solution picks activity X over our greedy choice G (where G ends earlier), we can swap G in for X without losing any other activity — proving greedy is at least as good as OPT.
Activities: [(1,3),(2,4),(3,5),(0,6),(5,7),(8,9),(5,9),(6,10),(8,11),(2,13),(12,14)] Sorted by end: [(1,3),(2,4),(3,5),(0,6),(5,7),(8,9),(5,9),(6,10),(8,11),(2,13),(12,14)] Greedy picks: (1,3) → (3,5) → (5,7) → (8,9) → (12,14) = 5 activities ✓
Python · Activity Selection — O(n log n)
def max_activities(activities):
    # activities = [(start, end), ...]
    activities.sort(key=lambda x: x[1])   # sort by END TIME

    selected = [activities[0]]             # always take the first (ends soonest)
    last_end = activities[0][1]

    for start, end in activities[1:]:
        if start >= last_end:              # no overlap with last selected
            selected.append((start, end))
            last_end = end

    return selected

# Interval Scheduling: Minimum Removals (LeetCode 435)
def erase_overlap_intervals(intervals):
    intervals.sort(key=lambda x: x[1])
    count = 0
    last_end = intervals[0][1]
    for start, end in intervals[1:]:
        if start < last_end:               # overlap! remove current (greedy: keep smaller end)
            count += 1
        else:
            last_end = end
    return count
📅 Activity Selection — sort by END time, then greedily keep non-overlapping events
selected (start ≥ last_end) rejected (overlaps) last_end boundary
Scanning in end-time order, keep an event only if it starts at/after last_end. Earliest finishers free up the most future time — so this simple rule is provably optimal. ▶ loops every 11s.
🎬 Problem 2: Jump Game — Can You Reach the End?

Array where each element = max jump length from that position. Classic greedy with a "furthest reach" tracker.

💡
The Insight: Track the maximum index you can reach so far. At each position i, update max_reach = max(max_reach, i + arr[i]). If i ever exceeds max_reach, you're stuck — no path exists. This is greedy because we always maximize our reach without needing to backtrack.
arr = [2,3,1,1,4] i=0: max_reach = max(0, 0+2) = 2 i=1: max_reach = max(2, 1+3) = 4 → already past end (len=5-1=4) → True ✓ arr = [3,2,1,0,4] i=0: max_reach = 3 i=1: max_reach = max(3,1+2) = 3 i=2: max_reach = max(3,2+1) = 3 i=3: max_reach = max(3,3+0) = 3 i=4: 4 > max_reach=3 → False ✓
Python · Jump Game I & II — O(n)
# Jump Game I: Can you reach last index?
def can_jump(arr):
    max_reach = 0
    for i, jump in enumerate(arr):
        if i > max_reach: return False    # stuck — can't reach position i
        max_reach = max(max_reach, i + jump)
        if max_reach >= len(arr) - 1: return True
    return True

# Jump Game II: Minimum jumps to reach end
def jump(arr):
    jumps = 0
    curr_end = 0      # end of current "jump level"
    farthest = 0      # furthest reachable within current level
    for i in range(len(arr) - 1):
        farthest = max(farthest, i + arr[i])
        if i == curr_end:    # must jump — level exhausted
            jumps += 1
            curr_end = farthest
    return jumps
# arr = [2,3,1,1,4]:  0→1(jump+1=1), 1→4(jump+1=2) → answer: 2 jumps ✓
🦘 Jump Game — track the furthest index reachable so far (arr = [2,3,1,1,4])
current index i reachable region (max_reach)
At each index extend max_reach = max(max_reach, i + arr[i]). As long as i ≤ max_reach you keep moving; the green region swallows the finish at i=1. (If i ever passes the green edge, you're stuck.) ▶ loops every 9s.
🎬 Problem 3: Fractional Knapsack — Greedy Works Here, 0/1 Does NOT

Understanding this distinction is critical — it shows when greedy is provably optimal vs dangerously wrong.

The Key Distinction

Fractional Knapsack: You can take parts of items

Greedy by value-per-weight ratio is optimal here. Take as much of the highest ratio item as fits, then next, etc.

0/1 Knapsack: You must take whole items or none. Greedy fails here! Example: capacity=10, items=[(weight=5,value=10),(weight=6,value=12),(weight=7,value=14)]. Greedy by ratio picks item 1 (ratio=2) + item 2 (ratio=2) = weight=11 > 10, can't fit both. So picks item 1 only = value=10. But optimal is item 2+something = actually item 1+item 3 = weight=12 > 10. Hmm, optimal is item 3 alone = 14 > 10. Greedy missed it. Use DP for 0/1 Knapsack.

Python · Fractional Knapsack — O(n log n)
def fractional_knapsack(items, capacity):
    # items = [(weight, value), ...]
    items.sort(key=lambda x: x[1] / x[0], reverse=True)  # sort by value/weight DESC

    total_value = 0.0
    for weight, value in items:
        if capacity >= weight:
            total_value += value
            capacity -= weight
        else:
            # take fraction of remaining capacity
            total_value += value * (capacity / weight)
            break               # knapsack full

    return total_value

# Trace: items=[(10,60),(20,100),(30,120)], capacity=50
# Ratios: 6.0, 5.0, 4.0 → sorted: same order
# Take item1: full (10), value=60, remaining=40
# Take item2: full (20), value=100, remaining=20
# Take item3: fraction 20/30, value=120*(20/30)=80
# Total = 60 + 100 + 80 = 240.0 ✓
🎒 Fractional Knapsack — fill by best value/weight ratio first (capacity = 50)
highest ratio (taken first) next ratio fractional fill of remainder
Sort by value/weight descending, then pour items in until the bag is full — splitting only the last one. Because value scales linearly with the fraction taken, grabbing the densest value first is provably optimal. ▶ loops every 10s.
🎬 Problem 4: Gas Station — Circular Tour

Find the starting gas station from which you can complete a full circular trip. The greedy insight here is beautiful.

💡
Two observations that make this greedy:
1. If total gas ≥ total cost, a solution always exists (single global feasibility condition).
2. If you run out of gas starting from station i, you also run out starting from any station between i and the failure point. So restart from the failure point + 1.
Why restart at failure+1? All stations between start and failure have already proven they can't be the starting point (they all accumulated negative surplus). The next untested station is failure+1.
gas=[1,2,3,4,5], cost=[3,4,5,1,2] net = [-2,-2,-2,3,3] (gas[i]-cost[i]) cumsum: -2,-4,-6,-3,0 ← dip below 0 at i=2 Restart from i=3: cumsum=0,3 → valid → answer: start=3 ✓
Python · Gas Station — O(n)
def can_complete_circuit(gas, cost):
    if sum(gas) < sum(cost): return -1   # total deficit → no solution

    start = 0
    tank = 0
    for i in range(len(gas)):
        tank += gas[i] - cost[i]
        if tank < 0:                     # ran out starting from current 'start'
            start = i + 1              # greedy: new candidate start
            tank = 0
    return start
⛽ Gas Station — when the tank goes negative, restart at the next station
tank negative (start fails) restart at failure + 1 tank stays ≥ 0 (valid start)
Every station from an old start that dipped negative is disqualified at once, so jump straight to failure + 1. One left-to-right pass finds the answer — and if total gas ≥ total cost, a valid start is guaranteed to exist. ▶ loops every 11s.

Meeting Rooms II — Minimum Rooms
Problem (LeetCode 253): Given n meeting time intervals [start, end], find the minimum number of conference rooms required.
Greedy + Min-Heap of end times: Sort by start. Use a min-heap tracking end times of ongoing meetings. For each new meeting: if the earliest-ending meeting is done (end ≤ current start), reuse its room (pop from heap). Either way, push current meeting's end time. Heap size at the end = max rooms used.
🔍 Step-by-Step Trace
intervals=[(0,30),(5,10),(15,20)] Sort by start: [(0,30),(5,10),(15,20)] heap=[] Process (0,30): heap empty → new room. heap=[30] rooms=1 Process (5,10): heap[0]=30 > 5 → new room. heap=[10,30] rooms=2 Process (15,20): heap[0]=10 ≤ 15 → reuse! pop 10, push 20. heap=[20,30] rooms=2 max_rooms = max heap size = 2 ✓ Overlap timeline: 0──────────────────────────────30 5─────10 15──20 t=0: 1 room | t=5: 2 rooms | t=10: 1 room
💻 Solution
Python · Meeting Rooms II
import heapq
def min_meeting_rooms(intervals):
    if not intervals: return 0
    intervals.sort(key=lambda x: x[0])  # sort by start
    heap = []  # min-heap of end times

    for start, end in intervals:
        if heap and heap[0] <= start:
            heapq.heapreplace(heap, end)  # reuse earliest-ending room
        else:
            heapq.heappush(heap, end)     # need a new room
    return len(heap)  # = max concurrent meetings
⌛ Complexity
Time: O(n log n) Space: O(n)

Task Scheduler — Formula Approach
Problem (LeetCode 621): Arrange CPU tasks with cooldown n. Minimum total time. Tasks labeled A-Z, same task must wait n intervals between executions.
Greedy insight: The most frequent task determines the floor. It creates (max_freq-1) complete frames of (n+1) slots plus a final partial frame. Fill gaps with other tasks or idle. If we have enough tasks to fill all gaps, answer is just len(tasks). Otherwise, it's the frame formula.
🔍 Step-by-Step Trace
tasks=['A','A','A','A','B','B','C'] n=2 freq: A=4, B=2, C=1 max_freq=4, count_of_max=1 (only A) Frames of width (n+1)=3: [A,B,C] [A,B,_] [A,_,_] [A] ^frame1 ^frame2 ^frame3 ^last formula = (4-1)*(2+1)+1 = 9+1 = 10 len(tasks)=7 < 10, so answer=10 Proof: we have 3 complete frames needing 3 slots each: slots needed = (max_freq-1)*(n+1) = 9 last frame = 1 slot (just A) total = 10 (3 idle slots filled = A,B,C + A,B,idle + A,idle,idle + A) When len(tasks) > formula (dense): answer=len(tasks) (no idle needed)
💻 Solution
Python · Task Scheduler
from collections import Counter
def least_interval(tasks, n):
    freq = Counter(tasks)
    max_freq = max(freq.values())
    max_count = sum(1 for f in freq.values() if f == max_freq)
    return max((max_freq-1)*(n+1) + max_count, len(tasks))
⌛ Complexity
Time: O(n) Space: O(1)

Partition Labels
Problem (LeetCode 763): String s. Partition it into as many parts as possible so each letter appears in at most one part. E.g. 'ababcbacadefegdehijhklij' → [9,7,8].
Greedy — extend until self-contained: Pre-compute last[c] = last index where char c appears. Scan left-right. Track current partition's rightmost boundary (max of last[c] for all c seen). When index i reaches boundary → close partition, start new.
🔍 Step-by-Step Trace
s = "ababcbacadefegde" last = {a:8, b:5, c:7, d:14, e:15, f:11, g:13} Scan: i=0 'a': boundary=max(0,last[a]=8)=8 i=1 'b': boundary=max(8,last[b]=5)=8 i=2 'a': boundary=max(8,8)=8 i=3 'b': boundary=max(8,5)=8 i=4 'c': boundary=max(8,last[c]=7)=8 i=5 'b': boundary stays 8 i=6 'a': boundary stays 8 i=7 'c': boundary stays 8 i=8: i==boundary → PARTITION! size=9 (0..8) Reset: start=9, boundary=8 i=9 'd': boundary=max(9,last[d]=14)=14 i=10 'e': boundary=max(14,last[e]=15)=15 ... i=15: i==boundary → PARTITION! size=7 Result: [9,7,...] ✓
💻 Solution
Python · Partition Labels
def partition_labels(s):
    last = {c: i for i, c in enumerate(s)}  # last occurrence of each char
    sizes = []
    start = boundary = 0

    for i, c in enumerate(s):
        boundary = max(boundary, last[c])   # extend partition if needed
        if i == boundary:
            sizes.append(i - start + 1)     # partition complete
            start = i + 1
    return sizes
⌛ Complexity
Time: O(n) Space: O(1) — 26 char alphabet

Minimum Number of Arrows to Burst Balloons
Problem (LeetCode 452): Balloons are horizontal ranges [x_start, x_end]. An arrow shot at x pops all balloons where x_start ≤ x ≤ x_end. Find minimum arrows to pop all.
Greedy — shoot at the end of the first balloon: Sort by end coordinate. Shoot an arrow at the first balloon's end. This arrow pops all other balloons that overlap at that end. Skip all popped balloons, repeat for next unpopped balloon.
🔍 Step-by-Step Trace
balloons=[(10,16),(2,8),(1,6),(7,12)] Sort by end: [(1,6),(2,8),(7,12),(10,16)] Arrow 1 at x=6: pops (1,6)✓ and (2,8)? 2<=6<=8 ✓ BOTH popped arrows=1, next unpopped=(7,12) Arrow 2 at x=12: pops (7,12)✓ and (10,16)? 10<=12<=16 ✓ BOTH popped arrows=2, done Total arrows = 2 ✓ Why shoot at END (not start)? Shooting at end maximizes overlap: it's the rightmost point that MUST be within the interval, and any balloon with start <= end is also popped.
💻 Solution
Python · Minimum Arrows
def find_min_arrow_shots(points):
    if not points: return 0
    points.sort(key=lambda x: x[1])   # sort by end
    arrows = 1
    arrow_pos = points[0][1]               # shoot at first balloon's end

    for start, end in points[1:]:
        if start > arrow_pos:             # not popped by last arrow
            arrows += 1
            arrow_pos = end               # shoot at this balloon's end
    return arrows
⌛ Complexity
Time: O(n log n) Space: O(1)

Candy Distribution
Problem (LeetCode 135): n children in a line with ratings. Rules: (1) Each child gets ≥1 candy. (2) Children with higher rating than adjacent neighbors must get more candy. Minimize total candies.
Two-pass greedy: Pass 1 (left-to-right): if rating[i] > rating[i-1], give one more than left. Pass 2 (right-to-left): if rating[i] > rating[i+1], candy[i] must be > candy[i+1] — take max. Both passes together satisfy both neighbor constraints.
🔍 Step-by-Step Trace
ratings = [1, 0, 2] Pass 1 (left→right): satisfy 'higher than left neighbor' candies = [1,1,1] i=1: 0 < 1, no change. candies=[1,1,1] i=2: 2 > 0, candies[2]=candies[1]+1=2. candies=[1,1,2] Pass 2 (right→left): satisfy 'higher than right neighbor' i=1: 0 < 2, no change. candies=[1,1,2] i=0: 1 > 0, candies[0]=max(1,candies[1]+1)=max(1,2)=2 candies=[2,1,2] Total = 2+1+2 = 5 ✓ ratings = [1,2,2]: P1: [1,2,1] P2: [1,2,1] → sum=4 Middle child has same rating as right, so no increase needed ✓
💻 Solution
Python · Candy Distribution
def candy(ratings):
    n = len(ratings)
    candies = [1] * n

    # Pass 1: left → right
    for i in range(1, n):
        if ratings[i] > ratings[i-1]:
            candies[i] = candies[i-1] + 1

    # Pass 2: right → left (take max to satisfy both directions)
    for i in range(n-2, -1, -1):
        if ratings[i] > ratings[i+1]:
            candies[i] = max(candies[i], candies[i+1] + 1)

    return sum(candies)
⌛ Complexity
Time: O(n) Space: O(n)
