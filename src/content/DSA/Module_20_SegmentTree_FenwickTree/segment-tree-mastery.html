Real World Story — Gaming Leaderboard

You run a competitive gaming platform. 1 million players. Scores update 100,000 times per second.

Your operations: (1) Update player score and (2) Sum of scores in rank range [L, R].

Naive array: Update = O(1). Sum of range = O(n) scan. With 100,000 updates and queries per second, you're doing 100,000 × 1,000,000 = 10¹¹ operations per second. Servers melt.

Prefix sum array: Sum = O(1). But update = O(n) to recompute all affected prefix sums. Same problem.

Segment Tree / Fenwick Tree: Both update AND query in O(log n). With n=1,000,000: log₂(1M) = 20. Both operations cost 20 steps. That's 10,000× faster than the naive approach.

The segment tree achieves this by organizing data as a tree where each node stores the aggregate of its subtree. An update flows up the tree (O(log n) nodes), and a query combines O(log n) precomputed ranges.

Segment Tree: The Core Insight

Array [1, 3, 5, 7, 9, 11] stored as a tree:

[36] (sum 0-5) / \
[9] (0-2) [27] (3-5)
/ \ / \
[4](0-1) [5](2) [16](3-4) [11](5)
/ \ / \
[1] [3] [7] [9]
Sum(1, 4) = arr[1]+arr[2]+arr[3]+arr[4]. Instead of scanning 4 elements, we combine [3] + [5] + [7] = node(0-1)'s right + node(2) + node(3)'s left. Just 3 lookups!

Python · Segment Tree — Sum with Point Updates
class SegmentTree:
    def __init__(self, arr):
        self.n = len(arr)
        self.tree = [0] * (4 * self.n)   # 4n is safe upper bound
        self._build(arr, 1, 0, self.n - 1)

    def _build(self, arr, node, start, end):
        if start == end:
            self.tree[node] = arr[start]  # leaf: store element
        else:
            mid = (start + end) // 2
            self._build(arr, 2*node,   start, mid)     # build left child
            self._build(arr, 2*node+1, mid+1, end)     # build right child
            self.tree[node] = self.tree[2*node] + self.tree[2*node+1]  # merge

    def update(self, idx, val, node=1, start=None, end=None):
        if start is None: start, end = 0, self.n - 1
        if start == end:
            self.tree[node] = val; return   # update leaf
        mid = (start + end) // 2
        if idx <= mid: self.update(idx, val, 2*node,   start, mid)
        else:          self.update(idx, val, 2*node+1, mid+1, end)
        self.tree[node] = self.tree[2*node] + self.tree[2*node+1]  # propagate up

    def query(self, l, r, node=1, start=None, end=None):
        if start is None: start, end = 0, self.n - 1
        if r < start or end < l: return 0        # no overlap: identity
        if l <= start and end <= r: return self.tree[node]  # full overlap: use cached
        mid = (start + end) // 2               # partial overlap: recurse both
        return self.query(l, r, 2*node, start, mid) + self.query(l, r, 2*node+1, mid+1, end)

🧠 Intuition — why a range query touches only O(log n) nodes

A segment tree pre-computes the aggregate of every power-of-two-aligned block and stacks them in a binary tree: the root covers the whole array, its children each cover a half, and so on down to single elements. The magic is that any range [L,R], no matter how it's positioned, can be tiled by at most 2·log n of these pre-computed blocks. The query walks down from the root and classifies each node three ways: fully inside the range → return its cached value and stop (don't go deeper); fully outside → return the identity (0 for sum); partially overlapping → recurse into both children. Because a node only splits when the range boundary cuts through it, and there are only two boundaries (L and R), the recursion fans out logarithmically. Updates run the same skeleton in reverse: change a leaf, then re-merge each ancestor on the way up.

🟢 Full overlap → use the cache

If a node's segment sits entirely inside [L,R], its stored aggregate is exactly what you need — return it and never descend. This is what caps the work at O(log n).

🔵 Partial overlap → recurse

The range boundary cuts this node. Split into both children and let each decide. Only nodes straddling L or R ever do this.

🟠 No overlap → identity

Disjoint from [L,R] → contribute the neutral element (0 for sum, +∞ for min). Prunes whole subtrees instantly.

🟣 Update propagates up

Set the leaf, then walk to the root re-computing node = left + right. Exactly one root-to-leaf path changes → O(log n).

query(L,R) visits ≤ 2·log₂ n nodes · full-overlap nodes are returned whole, never split
point update re-merges one root-to-leaf path → O(log n)
🌳 query(1, 4) on [1,3,5,7,9,11] — tiled by just 3 cached nodes = 24
full overlap → counted, stop partial → recurse no overlap → 0
The query never visits leaves 3 or 4 — node 3–4 is fully inside [1,4] so its cached 16 is used directly. Three cached nodes tile the whole range. ▶ loops every 12s.
⚡ Fenwick Tree (BIT) — Simpler, More Space Efficient
The Key Trick: i & (-i)

i & (-i) gives the lowest set bit of i. This is used for both navigation:
— Update: add i & (-i) to i (move to the next index that covers i)
— Query: subtract i & (-i) from i (peel off the last covered range)

Example: 6 = 110, 6 & (-6) = 010 = 2. So BIT[6] covers range [5,6] (length=2). Update 6 → update 8 (next coverage). Query prefix 6 → sum BIT[6]+BIT[4] (peeling 2, then 4).

Python · Fenwick Tree (BIT) — Prefix Sums
class BIT:
    def __init__(self, n):
        self.n = n
        self.tree = [0] * (n + 1)        # 1-indexed

    def update(self, i, delta):          # add delta to position i (1-indexed)
        while i <= self.n:
            self.tree[i] += delta
            i += i & (-i)                 # move to next responsible index

    def prefix_sum(self, i):             # sum of [1..i]
        s = 0
        while i > 0:
            s += self.tree[i]
            i -= i & (-i)                 # move to parent
        return s

    def range_sum(self, l, r):           # sum of [l..r] (1-indexed)
        return self.prefix_sum(r) - self.prefix_sum(l - 1)

# Build from array: O(n log n)
# bit = BIT(6)
# for i,v in enumerate([1,3,5,7,9,11], 1): bit.update(i, v)
# bit.range_sum(2, 5) → 3+5+7+9 = 24 ✓
⚙️ Fenwick prefix_sum(6) — peel off the lowest set bit, one cached block at a time
BIT[6] block [5,6] BIT[4] block [1,4] i & (-i) = lowest set bit
Each BIT[i] stores the sum of a block whose length is i & (-i) (the lowest set bit). A prefix sum repeatedly subtracts that bit, hopping across non-overlapping blocks that exactly tile [1..i]. ▶ loops every 10s.

Range Sum Query — Mutable
Problem (LeetCode 307): Implement NumArray class with update(i, val) and sumRange(l, r). Both operations should be O(log n). Naive array: update O(1), sum O(n). Prefix sum: sum O(1), update O(n).
Fenwick Tree (BIT): An array where each index stores the sum of a specific range of the original array, determined by the lowest set bit. update and query both take O(log n) by traversing these bit-based ranges.
🔍 Step-by-Step Trace

💻 Solution
Python · Range Sum Query Mutable (BIT)
class NumArray:
    def __init__(self, nums):
        self.n = len(nums)
        self.bit = [0] * (self.n + 1)
        for i, v in enumerate(nums): self._update(i+1, v)

    def _update(self, i, delta):    # 1-indexed
        while i <= self.n:
            self.bit[i] += delta
            i += i & (-i)          # next responsible index

    def _prefix(self, i):          # prefix sum [1..i]
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & (-i)          # parent in BIT tree
        return s

    def update(self, i, val):
        delta = val - (self._prefix(i+1) - self._prefix(i))
        self._update(i+1, delta)

    def sumRange(self, l, r):
        return self._prefix(r+1) - self._prefix(l)
⌛ Complexity
Time: O(log n) per op Space: O(n)

Count of Smaller Numbers After Self
Problem (LeetCode 315): For each element in array, count how many elements to its right are smaller. E.g. [5,2,6,1] → [2,1,1,0].
BIT + coordinate compression: Process elements right-to-left. For each element, query BIT for sum at (element-1) = count of smaller elements already inserted (from the right). Then insert element into BIT. Coordinate compress to handle any value range.
🔍 Step-by-Step Trace
nums = [5, 2, 6, 1] Coordinate compress: {1:1, 2:2, 5:3, 6:4} BIT size = 4 Process right-to-left: i=3, val=1 (mapped=1): query(0) = 0 → counts[3]=0 update(1, +1) → BIT[1]+=1 i=2, val=6 (mapped=4): query(3) = count(1,2,3 in BIT) = 1 (only 1 is there) counts[2]=1 update(4, +1) i=1, val=2 (mapped=2): query(1) = count(1 in BIT) = 1 (only 1 is there) counts[1]=1 update(2, +1) i=0, val=5 (mapped=3): query(2) = count(1,2 in BIT) = 2 (vals 1 and 2) counts[0]=2 update(3, +1) counts = [2, 1, 1, 0] ✓
💻 Solution
Python · Count of Smaller Numbers After Self
def count_smaller(nums):
    # Coordinate compress
    sorted_unique = sorted(set(nums))
    rank = {v: i+1 for i, v in enumerate(sorted_unique)}
    n = len(sorted_unique)
    bit = [0] * (n+1)

    def update(i):
        while i <= n: bit[i] += 1; i += i & (-i)
    def query(i):
        s = 0
        while i > 0: s += bit[i]; i -= i & (-i)
        return s

    result = []
    for v in reversed(nums):
        r = rank[v]
        result.append(query(r-1))   # count smaller elements already inserted
        update(r)
    return result[::-1]
⌛ Complexity
Time: O(n log n) Space: O(n)
