🎬 Rabin-Karp — Rolling Hash for Pattern Matching
Real World Story — DNA Sequencing

You're searching for a known genetic marker (pattern) in a 3-billion-character genome (text). Comparing character by character at every position takes O(n*m) — too slow.

Rabin-Karp's insight: Compute a hash of the pattern and a rolling hash of the current window. Compare hashes first (O(1)). Only do character comparison if hashes match. With a good hash function, collisions are rare, making average time O(n+m).

Rolling hash: hash(window[i+1..i+m]) can be computed from hash(window[i..i+m-1]) in O(1) by removing the leftmost character and adding the rightmost. Like sliding a window over hash values instead of characters.

Python · Rabin-Karp — O(n+m) average
def rabin_karp(text, pattern):
    n, m = len(text), len(pattern)
    BASE, MOD = 26, 10**9 + 7
    if m > n: return []

    # Precompute BASE^(m-1) % MOD
    power = pow(BASE, m-1, MOD)

    # Initial hashes
    pat_hash = win_hash = 0
    for i in range(m):
        pat_hash = (pat_hash * BASE + ord(pattern[i])) % MOD
        win_hash = (win_hash * BASE + ord(text[i]))    % MOD

    matches = []
    if win_hash == pat_hash and text[:m] == pattern:
        matches.append(0)

    for i in range(1, n-m+1):
        # Rolling hash: remove left char, add right char
        win_hash = (win_hash - ord(text[i-1]) * power) % MOD
        win_hash = (win_hash * BASE + ord(text[i+m-1])) % MOD
        if win_hash == pat_hash and text[i:i+m] == pattern:  # verify on match
            matches.append(i)

    return matches
🎞️ See it: the hash window rolling across the text

Searching for pattern "ABA" in "ABCABA". The window slides one step at a time; each move updates the hash in O(1) by dropping the left char and adding the right. Hashes are compared first — only a hash match triggers a character check.

O(n + m) average. The hash comparison is O(1); a full character check happens only on a hash hit, so spurious collisions cost little. ▶ loops every 8s.
current window hash + chars match text char
🎬 A* Search — Heuristic Shortest Path
A* vs Dijkstra

Dijkstra: Explores all neighbors by cost. No direction preference. Guaranteed optimal.
A*: Adds a heuristic h(n) estimating distance to goal. Priority = g(n) + h(n) where g(n) = actual cost so far. With admissible heuristic (never overestimates), A* is optimal and explores far fewer nodes.

Common heuristics: Manhattan distance (grids, no diagonals), Euclidean distance (grids with diagonals), 0 (= Dijkstra). Key: must never overestimate.

Python · A* Search on Grid
import heapq

def a_star(grid, start, goal):
    rows, cols = len(grid), len(grid[0])
    def h(r, c): return abs(r-goal[0]) + abs(c-goal[1])   # Manhattan distance

    dist = {start: 0}
    heap = [(h(*start), 0, start)]  # (f = g+h, g, position)
    prev = {}

    while heap:
        f, g, pos = heapq.heappop(heap)
        if pos == goal:                    # reconstruct path
            path = []
            while pos in prev: path.append(pos); pos = prev[pos]
            return path[::-1]
        if g > dist.get(pos, float('inf')): continue  # stale
        r, c = pos
        for dr, dc in [(0,1),(0,-1),(1,0),(-1,0)]:
            nr, nc = r+dr, c+dc
            if 0 <= nr < rows and 0 <= nc < cols and grid[nr][nc] != 1:
                ng = g + 1
                if ng < dist.get((nr,nc), float('inf')):
                    dist[(nr,nc)] = ng; prev[(nr,nc)] = pos
                    heapq.heappush(heap, (ng + h(nr,nc), ng, (nr,nc)))
    return []  # no path
🎞️ See it: A* expanding toward the goal, not in all directions

Because priority is f = g + h, A* favours cells whose heuristic h points at the goal — so it explores a narrow cone instead of a full circle (as plain Dijkstra would), then walks the reconstructed path.

Optimal with an admissible heuristic (one that never overestimates). Manhattan distance is the classic choice on a 4-directional grid. ▶ loops every 9s.

🎬 Bloom Filter — Probabilistic Set Membership
Real World Story — Spam Detection

Email provider wants to check if an email address is in a 10 billion spam list. A regular hash set uses 100GB. A Bloom filter uses 12GB.

Bloom Filter: A probabilistic data structure. Uses k hash functions and a bit array. To add element: set k bits. To check element: if ALL k bits are set → "probably in set". If ANY bit is clear → "definitely not in set".

False positives possible (might say "in set" for something not added). False negatives impossible (if in set, will always be found).

Used by: Redis, HBase, Cassandra, Chrome (safe browsing), Medium (already-read articles).

Python · Bloom Filter Implementation
import hashlib

class BloomFilter:
    def __init__(self, size, k_hashes):
        self.size = size
        self.k = k_hashes
        self.bits = [0] * size

    def _positions(self, item):
        positions = []
        for i in range(self.k):
            h = int(hashlib.md5(f"{item}{i}".encode()).hexdigest(), 16)
            positions.append(h % self.size)
        return positions

    def add(self, item):
        for pos in self._positions(item):
            self.bits[pos] = 1

    def contains(self, item):
        return all(self.bits[pos] for pos in self._positions(item))
        # True = probably in set | False = definitely NOT in set
🎞️ See it: setting bits on insert, then a "definitely not" on lookup

Insert "cat" → its 3 hash functions set bits 2, 6, 9. Later we query "dog": its hashes point at 2, 6, 10. Bit 10 is still 0 → "dog" is definitely not in the set (no false negatives, ever).

O(k) per operation, O(m) bits. No false negatives — a clear bit guarantees the item was never added. ▶ loops every 9s.
bit set to 1 probe on query bit 0 → absent

Rabin-Karp — Repeated DNA Sequences
Problem (LeetCode 187): Given a DNA string, find all 10-letter-long sequences that appear more than once. E.g. 'AAAAACCCCCAAAAACCCCCCAAAAAGGGTTT' → ['AAAAACCCCC','CCCCCAAAAA'].
Rolling hash (Rabin-Karp): Use a hash set to store seen 10-char substrings. Slide a window of 10 across the string, compute hash of each window. If hash already seen, record the substring. Rolling hash updates in O(1) by removing the leftmost char and adding the rightmost.
🔍 Step-by-Step Trace
s = "AAAAACCCCCAAAAACCCCCCAAAAAGGGTTT" Slide window of size 10: s[0:10] = "AAAAACCCCC" → hash → add to seen = {h1} s[1:11] = "AAAACCCCCA" → hash → add to seen s[2:12] = "AAACCCCCAA" → ... ... s[10:20] = "AAAAACCCCC" → hash found! → add to result Rolling hash: hash(s[i+1:i+11]) = (hash(s[i:i+10]) - val(s[i])*base^9) * base + val(s[i+10]) Each update: O(1) math instead of O(10) string hashing Result: ["AAAAACCCCC", "CCCCCAAAAA"] ✓ Interview note: Rabin-Karp shines for multiple pattern search. Single pattern: KMP is simpler; multiple patterns: Aho-Corasick.
💻 Solution
Python · Repeated DNA Sequences (Rolling Hash)
def find_repeated_dna(s):
    if len(s) <= 10: return []
    seen = set()
    repeated = set()
    for i in range(len(s)-9):
        sub = s[i:i+10]
        if sub in seen: repeated.add(sub)
        else: seen.add(sub)
    return list(repeated)

# Optimized: encode each char to 2 bits for faster hashing
def find_repeated_dna_opt(s):
    encode = {'A':0,'C':1,'G':2,'T':3}
    MASK = (1 << 20) - 1  # 20 bits for 10 chars × 2 bits each
    h = 0
    for c in s[:9]: h = ((h << 2) | encode[c]) & MASK
    seen = set(); repeated = set()
    for i in range(9, len(s)):
        h = ((h << 2) | encode[s[i]]) & MASK
        if h in seen: repeated.add(s[i-9:i+1])
        else: seen.add(h)
    return list(repeated)
⌛ Complexity
Time: O(n) Space: O(n)

Bloom Filter Design — System Design
Question: Design a system that can efficiently check if a username is 'definitely not taken' or 'possibly taken' when 100 million usernames exist. Exact lookup requires too much memory.
Bloom Filter: A probabilistic data structure with O(1) query and no false negatives. Use k independent hash functions mapping each item to positions in a bit array. Lookup: check all k positions — if ANY is 0, item is definitely absent. If all 1, item is probably present (small false positive rate).
🔍 Step-by-Step Trace
Bloom filter: bit_array = [0]*m (m=1000), k=3 hash functions Insert "alice": h1("alice")=42, h2("alice")=100, h3("alice")=571 Set bits[42]=bits[100]=bits[571]=1 Insert "bob": h1("bob")=10, h2("bob")=200, h3("bob")=42 (collision at 42!) Set bits[10]=bits[200]=1 (bits[42] already 1) Check "alice": bits[42]=1 ✓, bits[100]=1 ✓, bits[571]=1 ✓ → "possibly present" Check "charlie": h1("charlie")=5: bits[5]=0 → DEFINITELY ABSENT (no false negative) ✓ Check "dave": h1=10✓, h2=42✓, h3=100✓ → "possibly present" (FALSE POSITIVE!) "dave" was never inserted but all its positions happen to be 1 False positive rate ≈ (1-e^(-kn/m))^k With m=1B, k=7, n=100M → ~0.8% false positive rate
💻 Solution
Python · Bloom Filter Implementation
import hashlib
class BloomFilter:
    def __init__(self, size=1000000, num_hashes=7):
        self.size = size
        self.num_hashes = num_hashes
        self.bits = [0] * size

    def _positions(self, item):
        for seed in range(self.num_hashes):
            h = int(hashlib.md5(f'{seed}:{item}'.encode()).hexdigest(), 16)
            yield h % self.size

    def add(self, item):
        for pos in self._positions(item):
            self.bits[pos] = 1

    def might_contain(self, item):
        return all(self.bits[pos] for pos in self._positions(item))
    # False = definitely absent | True = probably present
⌛ Complexity
Time: O(k) per op — k hash functions Space: O(m) bit array

Implement LRU + LFU Cache Hybrid
Question: Design a cache that evicts the least frequently used item, and among same-frequency items, evicts the least recently used. This is LFU cache (LeetCode 460).
HashMap + doubly-linked list per frequency bucket: Main map: key→(value, freq). Freq map: freq→OrderedDict (maintains insertion order within same frequency). Track min_freq. On get/put: update frequency, move to higher bucket.
🔍 Step-by-Step Trace
capacity=2 put(1,1): freq_map={1: {1:1}}, key_freq={1:1}, min_freq=1 put(2,2): freq_map={1: {1:1, 2:2}}, key_freq={1:1,2:2}, min_freq=1 get(1): freq of key 1: 1→2 freq_map={1:{2:2}, 2:{1:1}}, min_freq=1 (bucket 1 still has 2) put(3,3): capacity reached, evict min_freq=1 bucket's LRU freq_map[1]={2:2}, oldest in freq 1 = key 2 → evict key 2 put(3,3): freq_map={1:{3:3}, 2:{1:1}}, min_freq=1 get(2): key 2 was evicted → return -1 get(3): freq of 3: 1→2, freq_map={2:{1:1,3:3}}, min_freq=2 put(4,4): evict from min_freq=2 → oldest key in freq 2 = key 1 put(4,4): min_freq=1
💻 Solution
Python · LFU Cache
from collections import OrderedDict
class LFUCache:
    def __init__(self, capacity):
        self.cap = capacity
        self.min_freq = 0
        self.key_val  = {}           # key → value
        self.key_freq = {}           # key → frequency
        self.freq_keys = {}          # freq → OrderedDict of keys

    def _touch(self, key):
        f = self.key_freq[key]
        self.key_freq[key] += 1
        self.freq_keys[f].pop(key)
        if not self.freq_keys[f]:
            del self.freq_keys[f]
            if self.min_freq == f: self.min_freq += 1
        self.freq_keys.setdefault(f+1, OrderedDict())[key] = True

    def get(self, key):
        if key not in self.key_val: return -1
        self._touch(key)
        return self.key_val[key]

    def put(self, key, value):
        if not self.cap: return
        if key in self.key_val:
            self.key_val[key] = value; self._touch(key); return
        if len(self.key_val) >= self.cap:
            evict = next(iter(self.freq_keys[self.min_freq]))
            self.freq_keys[self.min_freq].pop(evict)
            del self.key_val[evict], self.key_freq[evict]
        self.key_val[key] = value
        self.key_freq[key] = 1
        self.freq_keys.setdefault(1, OrderedDict())[key] = True
        self.min_freq = 1
⌛ Complexity
Time: O(1) per op Space: O(capacity)
