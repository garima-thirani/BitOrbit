🧠 The Story — The Phone Book
Real World Story — Binary Search Meets Tree

A sorted phone book lets you jump to the middle, decide "before or after?", and recurse on the right half.

BST materializes this process as a tree. Every node satisfies: all values in left subtree < node.val < all values in right subtree.

Searching: start at root. If target < current → go left. If target > current → go right. If equal → found. O(h) where h = tree height.

The catch: An unbalanced BST (e.g., inserting sorted data into a plain BST) degrades to a linked list: h = O(n), all operations O(n). Self-balancing trees (AVL, Red-Black, Python's sortedcontainers.SortedList) maintain h = O(log n) automatically.

Key BST Property: In-order = Sorted

In-order traversal of a BST always produces elements in sorted ascending order. This means:
— "Find kth smallest" = do in-order traversal, return kth element.
— "Validate BST" = check that in-order output is strictly increasing.
— "Convert BST to sorted array" = in-order traversal.

💡 The Deep Intuition — Binary Search, Frozen Into Pointers

You already know binary search on a sorted array: look at the middle, decide "too big or too small", throw away half. A BST is that idea made permanent. Instead of recomputing the middle every time, each node remembers who is smaller (left) and who is bigger (right). So a search is just "walk down, at each step throw away one whole subtree." The price you pay over an array is pointers; the reward is that insert and delete are also O(h) — no shifting elements like a sorted array forces.

🟡 The one invariant

For every node: everything in the left subtree is strictly smaller, everything in the right subtree is strictly larger. Not just the direct children — the whole subtree. This is the rule that makes "go left or go right" a correct decision and is exactly the rule a "validate BST" question tests.

🔵 Why O(h), not O(log n)

Each comparison drops you one level and discards a subtree, so the cost is the height h. In a balanced tree h ≈ log n — wonderful. But the tree's shape depends on insertion order; nothing in a plain BST forces balance.

🟣 The magic property

An in-order traversal (Left → Node → Right) always emits values in sorted order. This single fact answers a whole family of questions for free:

🟢 Deleting is the hard part

Removing a leaf or a one-child node is trivial. A two-child node can't just vanish — you replace its value with its in-order successor (the smallest value in its right subtree), then delete that successor, which always has at most one child.

🌍 Where BSTs (and their balanced cousins) run the world
  • Database indexes: the B-tree behind a SQL index is a fat, disk-friendly BST. WHERE age BETWEEN 30 AND 40 is a BST range walk — that's why an index makes the query fast.
  • The "guess my number 1–100" game: each "higher / lower" is one BST comparison; you win in ~7 guesses because log₂(100) ≈ 7.
  • A leaderboard or order book: "who is ranked just above me?" / "best ask price" is a successor query, kept fast by a balanced BST.
  • Auto-complete & spell-checkers: ordered maps (C++ std::map, Java TreeMap) are red-black trees — BSTs that rebalance on every insert.
  • File-system & memory allocators: finding the smallest free block ≥ a size is a BST lookup.
⚠️ The two classic traps

1. Validation by neighbours. Checking only left.val < node.val < right.val is wrong — a deep descendant can still violate the rule. You must carry a (low, high) bound down the tree. 2. The degenerate tree. Insert already-sorted data (1, 2, 3, 4, 5…) into a plain BST and it becomes a straight line: h = n, every operation O(n). That is why production code uses self-balancing trees, not a hand-rolled BST.

Python · BST — Search, Insert, Delete
class BST:
    def search(self, root, val):
        if not root or root.val == val: return root
        if val < root.val: return self.search(root.left,  val)
        return                   self.search(root.right, val)

    def insert(self, root, val):
        if not root: return TreeNode(val)
        if val < root.val: root.left  = self.insert(root.left,  val)
        else:              root.right = self.insert(root.right, val)
        return root

    def delete(self, root, val):
        if not root: return None
        if   val < root.val: root.left  = self.delete(root.left,  val)
        elif val > root.val: root.right = self.delete(root.right, val)
        else:  # found: 3 cases
            if not root.left:  return root.right   # no left child
            if not root.right: return root.left    # no right child
            # Two children: replace with in-order successor (min of right subtree)
            successor = root.right
            while successor.left: successor = successor.left
            root.val   = successor.val                # copy successor value
            root.right = self.delete(root.right, root.val)  # delete successor
        return root
🎬 search(7) — at every node, throw away an entire subtree
Each comparison halves the problem. Going left at the root instantly eliminates 10, 14 and everything under them — no need to look. In a balanced tree this is O(log n); the path length is exactly the tree's height. ▶ loops every 10s.
🎬 delete(3) with two children — promote the in-order successor
Why the successor? The smallest value in the right subtree is larger than everything on the left and smaller than everything else on the right — so dropping it into the hole keeps the BST invariant intact. And because it's the leftmost node, it has no left child, making its own deletion the easy one-child case. ▶ loops every 11s.

🎬 Validate BST & Kth Smallest
Python · Validate BST and Kth Smallest
# --- Validate BST (correct approach: pass min/max bounds) ---
def is_valid_bst(root, lo=float('-inf'), hi=float('inf')):
    if not root: return True
    if not (lo < root.val < hi): return False
    return (is_valid_bst(root.left,  lo, root.val) and
            is_valid_bst(root.right, root.val, hi))
# ⚠️ Common mistake: only checking root.val > root.left.val is WRONG.
# Counter-example:  [5, 1, 7, null, null, 6, null] → 6 is right child of 7 which
# is right child of 5... but 6 < 5 is impossible so this is invalid. Need the bounds!

# --- Kth Smallest: in-order + early exit ---
def kth_smallest(root, k):
    stack, count = [], 0
    while stack or root:
        while root: stack.append(root); root = root.left  # go left
        root = stack.pop()
        count += 1
        if count == k: return root.val    # found kth in-order = kth smallest
        root = root.right
🎬 In-order traversal — the BST "unrolls" itself into a sorted line
Left · Node · Right, recursively ⇒ 1·3·4·6·7·8·10·14, perfectly sorted. This is the single most useful BST fact: "kth smallest" stops at the kth box; "validate BST" just checks each box is bigger than the last; "two-sum in a BST" walks this line from both ends. ▶ loops every 12s.
🎯 Interview Questions — Detailed Walkthroughs

Convert Sorted Array to BST
Problem (LeetCode 108): Given an integer array sorted in ascending order, convert it to a height-balanced BST (where the depth of any two leaf nodes differs by at most 1).
Recursive mid-selection: Always pick the middle element of current range as the root. The left half becomes the left subtree, right half becomes the right subtree. This guarantees balance because each side has equal (or ±1) number of elements.
🔍 Step-by-Step Trace
nums = [-10,-3,0,5,9] Indices: 0,1,2,3,4 build(0,4): mid=2, root=0 build(0,1): mid=0, root=-10 build(0,-1): return None build(1,1): mid=1, root=-3 → leaf node -3 Left subtree: 0 → (-10) → (-3) (-10) -3 build(3,4): mid=3, root=5 build(3,2): return None build(4,4): mid=4, root=9 (leaf) Right subtree: 5 9 Final BST: 0 / -10 5 \ -3 9 Height: 3 (balanced ✓)
💻 Solution
Python · Sorted Array to BST
def sorted_array_to_bst(nums):
    def build(l, r):
        if l > r: return None
        mid = (l + r) // 2
        node = TreeNode(nums[mid])
        node.left  = build(l, mid-1)   # left half
        node.right = build(mid+1, r)   # right half
        return node
    return build(0, len(nums)-1)
⌛ Complexity
Time: O(n) Space: O(log n) stack
2
Inorder Successor in BST
Problem (LeetCode 285): Given a BST and a node p, find the in-order successor (smallest value greater than p.val).
Two cases: (1) If p has a right child, successor = leftmost node in right subtree. (2) If no right child, successor = the last ancestor where you went LEFT. Iterative traversal: if current node > target, save as candidate and go left; if ≤ target, go right.
🔍 Step-by-Step Trace
BST: 20 / 8 22 / 4 12 / 10 14 Find successor of p=8: Case 1: p=8 has right child (12). Leftmost of right subtree: 12 → 10 Successor = 10 ✓ Find successor of p=14: Case 2: p=14 has no right child. Traverse from root: node=20: 20>14 → candidate=20, go left node=8: 8<14 → go right (no candidate here, went right) node=12: 12<14 → go right (no candidate here) node=14: 14=14 → no right child, done Last candidate where we went left = 20 ✓
💻 Solution
Python · Inorder Successor in BST
def inorder_successor(root, p):
    successor = None
    node = root
    while node:
        if node.val > p.val:
            successor = node       # candidate: went left here
            node = node.left
        else:
            node = node.right      # p's successor must be further right
    return successor
⌛ Complexity
Time: O(h) where h=height Space: O(1)
