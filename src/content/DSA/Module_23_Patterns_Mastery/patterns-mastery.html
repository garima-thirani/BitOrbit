📚 Pattern 1: Two Pointers

Two indices walk the array in a coordinated way, turning an O(n²) brute force into a single O(n) sweep. The magic: each comparison lets you discard a whole region of the search space.

🎯 When To Reach For It

  • Array is sorted (or can be sorted)
  • Find a pair/triplet meeting a sum/difference condition
  • Reverse, partition, or dedupe in place
  • You smell a nested loop that recompares the same elements

🧩 The Core Idea

Place left at the start and right at the end. Compare what they point to against the target, then move the pointer that brings you closer to the answer.

Because the array is sorted, moving left right only increases the sum; moving right left only decreases it. Every move is a deliberate step toward the goal.

🔁 The Decision Rule

sum = a[left] + a[right]
if sum == target → found
if sum < target → left++ # need bigger
if sum > target → right-- # need smaller

🌱 Variations

  • Opposite ends: two-sum, container with most water
  • Same direction (fast/slow): remove duplicates, partition
  • Two arrays: merge sorted lists

See it: pointers converging on a target

Two Sum on sorted [2, 7, 11, 15], target = 18. Pointers start at the two ends and squeeze inward.

O(n) not O(n²) — each step throws away one element permanently, so the pointers meet after at most n moves. ▶ animation loops every 8s.
winning pair discarded
⚠️ Common Pitfall

Two pointers needs sorted order for the "move toward target" logic to be valid. If the problem needs original indices (like LeetCode's unsorted Two Sum), sorting destroys them — use a hash map instead. For 3Sum, remember to skip duplicates on both pointers to avoid repeated triplets.

📚 Pattern 2: Sliding Window

A contiguous window [left, right] slides across the array. Instead of recomputing each subarray from scratch, you incrementally add the entering element and remove the leaving one — O(n) total.

🎯 When To Reach For It

  • "Longest/shortest/optimal contiguous subarray/substring"
  • A constraint that grows as the window grows (sum ≥ k, ≤ k distinct chars)
  • Fixed-size window (max average of k elements)

🧩 The Core Idea

Grow the window by moving right. The moment the window violates the constraint, shrink it from the left until it's valid again. Record the best valid window along the way.

Each index enters once and leaves once → 2n moves → O(n).

🔁 The Loop Skeleton

for right in range(n):
  add(a[right])
  while invalid():
    remove(a[left]); left++
  best = max(best, right-left+1)

🌱 Two Flavors

  • Variable window: longest substring without repeats
  • Fixed window: slide size-k window, update on enter/exit
  • "At most K" trick: exactly(K) = atMost(K) − atMost(K−1)

See it: the window expanding & contracting

Longest substring without repeating chars in "abcabcbb". When a duplicate enters, left jumps past the previous occurrence.

Window never re-shrinks below valid — left only moves forward, right only moves forward. ▶ loops every 8s.
current window duplicate (forces shrink) not yet seen
⚠️ Common Pitfall

Decide when you record the answer. For "longest valid" you update the best after restoring validity. For "shortest valid" (min window) you update the best inside the shrink loop, while the window is still valid. Putting the update in the wrong place is the classic sliding-window bug.

📚 Pattern 3: Fast & Slow Pointers (Floyd's Cycle)

Two pointers move at different speeds through a linked list. If there's a loop, the fast one inevitably laps the slow one — like a faster runner catching a slower runner on a circular track.

🎯 When To Reach For It

  • Detect a cycle in a linked list
  • Find the middle node in one pass
  • Find the cycle's entry point
  • "Happy number", duplicate detection via functional graph

🧩 The Core Idea

slow takes 1 step, fast takes 2. If the list ends, no cycle. If they ever land on the same node, there's a cycle — because the gap shrinks by 1 each step inside the loop until it hits 0.

🔁 The Movement

slow = slow.next
fast = fast.next.next
if slow == fast → cycle!
# gap closes by 1 every step

🌱 Finding the Entry

After they meet, reset one pointer to head. Move both at speed 1. Where they meet again is the cycle start.

dist(head→start) ==
dist(meet→start)

See it: the gap closing inside the loop

A linked list with a cycle. slow (blue) and fast (amber) both enter the loop; fast gains on slow until they collide.

O(n) time, O(1) space. Fast gains 1 position per step; inside the loop the gap shrinks to 0 and they collide. ▶ loops every 8s.
slow (1 step) fast (2 steps) cycle entry
⚠️ Common Pitfall

Always check fast and fast.next before fast.next.next — otherwise you hit a null-pointer error at the end of a non-cyclic list. The two pointers must start aligned (both at head) for the entry-finding math to hold.

📚 Pattern 4: Merge Intervals

Given a pile of ranges, sort them by start time and sweep left-to-right, fusing any that touch or overlap. After sorting, overlaps can only happen with the immediately preceding interval — that's what makes one pass enough.

🎯 When To Reach For It

  • Calendar/meeting scheduling, booking conflicts
  • Merge or insert overlapping ranges
  • "Minimum rooms / max overlap" questions
  • Anything with [start, end] pairs

🧩 The Core Idea

Sort by start. Keep a "current" merged interval. For each next interval, if it starts before the current one ends, they overlap — stretch the current end. Otherwise, close out current and start fresh.

🔁 The Decision Rule

if next.start ≤ cur.end:
  cur.end = max(cur.end, next.end)
else:
  output(cur); cur = next

🌱 Variations

  • Insert interval: 3 phases — before, merge, after
  • Min meeting rooms: sweep-line on start/end events
  • Interval intersection: two-pointer on two lists

See it: sorted bars fusing together

Merge [[1,3],[2,6],[8,10],[9,12]]. After sorting, [1,3] and [2,6] overlap → [1,6]; [8,10] and [9,12] overlap → [8,12].

Sort once O(n log n), sweep once O(n). The sort guarantees an interval can only overlap its predecessor. ▶ loops every 8s.
input intervals merged result sweep line
⚠️ Common Pitfall

Decide whether touching intervals like [1,2] and [2,3] count as overlapping (≤ vs <). Forgetting to sort first breaks the entire invariant — without sorting, an interval could overlap one far earlier in the list.

[start, end] pairssort by startoverlap mergeschedulingsweep line
📚 Pattern 5: BFS (Level-Order Traversal)

Explore a graph or tree in concentric rings — all nodes at distance 1, then distance 2, and so on. Because BFS visits in order of distance, the first time it reaches a node is guaranteed to be the shortest path (in an unweighted graph).

🎯 When To Reach For It

  • Shortest path in an unweighted graph/grid
  • Level-order tree traversal
  • "Minimum steps/moves to reach a state"
  • Word ladder, rotting oranges, maze escape

🧩 The Core Idea

Use a queue (FIFO). Pop a node, push its unvisited neighbors. Mark nodes visited when enqueued (not dequeued) to avoid adding them twice. Process a whole level before the next.

🔁 The Loop Skeleton

q = deque([start]); seen = {start}
while q:
  node = q.popleft()
  for nb in neighbors(node):
    if nb not in seen:
      seen.add(nb); q.append(nb)

🌱 Level Tracking

To count distance, snapshot len(q) at the top of each iteration and process exactly that many nodes — that's one full level.

See it: concentric rings of distance

BFS from node A. Level 0 = {A}, level 1 = direct neighbors, level 2 = their neighbors. The first arrival at any node is its shortest distance.

Queue order = distance order. Shortest path is found the instant a node is first dequeued. ▶ loops every 8s.
source (dist 0) level 1 level 2
⚠️ Common Pitfall

Mark nodes visited when you enqueue, not when you dequeue. If you wait until dequeue, the same node can be pushed multiple times before being processed, blowing up memory and breaking the distance guarantee. BFS only gives shortest paths for unweighted edges — use Dijkstra when weights differ.

📚 Pattern 6: DFS & Recursion

Plunge down one path as far as it goes, then back up and try the next. DFS is the natural fit for "explore every possibility" problems — permutations, combinations, path-finding — because the call stack remembers exactly where to resume.

🎯 When To Reach For It

  • Enumerate all paths/subsets/permutations
  • Backtracking with constraints (N-Queens, Sudoku)
  • Connectivity, island counting, tree traversal
  • "Does a path exist?" / "collect all solutions"

🧩 The Core Idea

Choose → explore → un-choose. Make a decision, recurse deeper, then undo it so you can try the next option with a clean slate. The recursion tree's leaves are your candidate answers.

🔁 The Backtracking Template

def dfs(path, choices):
  if complete(path): record(path); return
  for c in choices:
    path.add(c) # choose
    dfs(path, remaining)# explore
    path.pop() # un-choose

🌱 Pruning

Cut branches early when they can't lead to a valid answer (e.g. skip used numbers, enforce start index for combinations to avoid duplicates). Pruning is what makes exponential search tractable.

See it: the recursion tree of choices

Permutations of [1,2,3]. Each level fixes one more position; each root-to-leaf path is one permutation. Backtracking pops the last choice to explore siblings.

Choose → explore → un-choose. Watch it commit to branch "1" completely before backtracking to "2". ▶ loops every 9s.
partial choice complete permutation
⚠️ Common Pitfall

Forgetting to undo the choice after recursing leaks state into sibling branches. For combinations (order doesn't matter) pass a start index so you never revisit earlier elements — otherwise you'll generate the same set in different orders.

all paths/subsetsbacktrackingchoose/explore/undorecursion treepruning
📚 Pattern 7: Topological Sort

Order tasks so every prerequisite comes before the task that needs it. You repeatedly take any task with no remaining prerequisites, "complete" it, and unlock its dependents. If you ever get stuck with tasks left, there's a cycle.

🎯 When To Reach For It

  • Course prerequisites, build/dependency order
  • "Is this schedule possible?" (cycle detection in a DAG)
  • Any "X must come before Y" constraints

🧩 The Core Idea (Kahn's)

Compute indegree (number of prerequisites) for every node. Start a queue with all indegree == 0 nodes. Pop one, append to order, and decrement its neighbors' indegrees — newly-zero ones join the queue.

🔁 The Loop

q = [n for n if indeg[n]==0]
while q:
  u = q.pop(); order.append(u)
  for v in adj[u]:
    indeg[v]--; if 0: q.add(v)

🌱 Cycle Check

If len(order) < n at the end, some nodes never reached indegree 0 → they're trapped in a cycle → no valid ordering.

See it: peeling off zero-indegree nodes

Course graph: 0→1, 0→2, 1→3, 2→3. Node 0 has no prereqs, so it goes first; finishing it unlocks 1 and 2; finishing both unlocks 3.

O(V + E). Each node is queued once when its last prerequisite clears. If the order is shorter than n, a cycle exists. ▶ loops every 8s.
indegree 0 (ready) still waiting
⚠️ Common Pitfall

Build edges in the correct direction: prerequisite → course. Mixing it up inverts the whole order. Always handle the cycle case explicitly — returning a partial order as if valid is a silent bug.

prerequisitesDAG orderingindegree / Kahn'scycle detectionbuild order
📚 Pattern 8: Binary Search

Halve the search space every step. The deeper insight: binary search isn't just for sorted arrays — it works on any monotonic predicate, a space that looks like [false, false, …, true, true], where you hunt the boundary.

🎯 When To Reach For It

  • Sorted array lookup, first/last occurrence
  • "Minimum k such that condition(k) is true"
  • Binary search on the answer (min capacity, min speed)
  • Rotated/monotonic structure

🧩 The Core Idea

Keep a range [lo, hi] that's guaranteed to contain the answer. Test the middle; the result tells you which half to discard. Repeat until the range is one element.

🔁 The Loop

while lo < hi:
  mid = (lo + hi) // 2
  if ok(mid): hi = mid # answer ≤ mid
  else: lo = mid + 1 # answer > mid

🌱 Boundary Discipline

Pick a template and stick to it: lo < hi with hi = mid finds the first true. Getting +1 placement wrong causes infinite loops or off-by-one — the #1 binary search bug.

See it: the range collapsing by half

Search for 9 in [1,3,5,7,9,11,13]. Each probe of the middle discards half the remaining candidates.

O(log n). 7 elements → at most 3 probes. A million elements → only 20. ▶ loops every 7.5s.
middle probe discarded half target found
⚠️ Common Pitfall

Use lo + (hi - lo) // 2 in languages where lo + hi can overflow. The loop condition (< vs ≤) must match how you shrink the bounds, or you loop forever. When binary-searching the answer, double-check the predicate is truly monotonic.

sorted / monotonicfirst/last truesearch the answerO(log n)boundary care
📚 Pattern 9: Heap & Priority Queue

A heap always hands you the smallest (or largest) element in O(log n), without keeping everything fully sorted. For "top K" problems, a size-K heap is the secret: you never store more than K elements, so it's O(n log k) instead of O(n log n).

🎯 When To Reach For It

  • "Top / Kth largest / smallest" elements
  • Merge K sorted lists/streams
  • Running median, scheduling by priority
  • Repeatedly grab the current min/max

🧩 The Core Idea

A binary heap is a complete tree where every parent ≤ its children (min-heap). The minimum is always the root. Push/pop re-bubble one element along a single root-to-leaf path → O(log n).

🔁 Top-K Trick

for x in nums:
  heappush(h, x)
  if len(h) > k: heappop(h)
# h holds the k largest; root = kth largest

🌱 Min vs Max

Python's heapq is a min-heap. For a max-heap, push -x (negate). For "k largest" keep a min-heap of size k so the weakest survivor is on top, ready to be evicted.

See it: array as a binary heap tree

A min-heap. The root is the global minimum. Popping it pulls the last element to the root, then it "sifts down" swapping with its smaller child until order is restored.

push/pop O(log n). Only one path from root to leaf is ever touched — the rest of the tree stays put. ▶ loops every 8s.
root = current min popped out heap node
⚠️ Common Pitfall

For "k largest" use a min-heap of size k (not a max-heap of everything) — that's what gives O(n log k). When pushing tuples for tie-breaking, include a unique counter so Python never tries to compare the payload objects directly.

top / Kth Kmerge K listsrunning medianO(n log k)min vs max heap
📚 Pattern 10: Union-Find (Disjoint Set)

A data structure for the question "are these two things in the same group?" — answered in near-constant time. Each set is a tree pointing to a representative root; merging groups is just hooking one root under another.

🎯 When To Reach For It

  • Count connected components
  • Detect a cycle in an undirected graph
  • Dynamically merge groups / equivalences
  • Kruskal's MST, account merging

🧩 The Core Idea

find(x) walks parent pointers up to the root (the group's id). union(a,b) finds both roots and links one under the other. Same root ⇒ same group.

🔁 The Two Ops

def find(x):
  if p[x]!=x: p[x]=find(p[x]) # compress
  return p[x]
def union(a,b): p[find(a)] = find(b)

🌱 Two Optimizations

Path compression (flatten the tree during find) + union by rank (attach smaller tree under bigger). Together they give near-O(1) amortized — α(n), inverse Ackermann.

See it: two components merging into one

Nodes 1-2-3 form one group, 4-5 another. A union(3,4) edge links the two roots, fusing everything into a single component.

Near O(1) per op with path compression + union by rank. If union connects two nodes already sharing a root, you've found a cycle. ▶ loops every 7s.
set root uniting edge member node
⚠️ Common Pitfall

Always union(find(a), find(b)) on the roots, not on a and b directly — otherwise you build long chains and lose the speed. For grid problems, map 2D cells to 1D ids with r*cols + c.

connected componentsundirected cyclepath compressionunion by rankdynamic merge
📚 Pattern 11: Trie (Prefix Tree)

A tree where every path from the root spells out a prefix. Shared prefixes share nodes, so "app", "apple", "apply" all reuse the same a→p→p branch. Prefix queries become a simple walk down the tree.

🎯 When To Reach For It

  • Autocomplete / prefix search
  • Dictionary word lookup, spell-check
  • "Does any word start with …?"
  • Word Search II, maximum XOR pair

🧩 The Core Idea

Each node holds a map of child letters and an isEnd flag. Inserting a word walks/creates one node per character. The final node is marked as a complete word.

🔁 Insert / Search

def insert(w):
  node = root
  for c in w: node = node.child[c]
  node.isEnd = True

🌱 Why It Wins

Lookup is O(L) in the word length — independent of how many words are stored. A hash set can't answer "all words with prefix X" efficiently; a trie does it by exploring one subtree.

See it: inserting "app" then "apple"

"app" creates the first three nodes (last marked end-of-word). "apple" reuses a→p→p and only adds l→e.

Insert/search O(L) in word length. Shared prefixes are stored exactly once — that's the whole point. ▶ loops every 8s.
node being walked end of word trie node
⚠️ Common Pitfall

"app" being a prefix of "apple" doesn't mean "app" is a stored word — you must check the isEnd flag, not just whether the path exists. For Word Search II, prune trie branches as words are found to avoid re-exploring.

prefix searchautocompleteisEnd flagshared prefixesO(L) lookup
📚 Pattern 12: Monotonic Stack

A stack kept in sorted order (increasing or decreasing). When a new element breaks the order, you pop — and each pop resolves an answer. It turns many "next greater/smaller element" problems from O(n²) into a single O(n) pass.

🎯 When To Reach For It

  • Next greater / next smaller element
  • Daily Temperatures, stock span
  • Largest rectangle in a histogram
  • Trapping rain water

🧩 The Core Idea

Push indices, keeping the stack monotonic. When the incoming value violates the order, pop: the popped element has just found its "next greater" (the incoming one). Each index is pushed and popped at most once.

🔁 The Loop

for i, x in enumerate(a):
  while st and a[st[-1]] < x:
    j = st.pop(); ans[j] = i - j
  st.append(i)

🌱 Why O(n)

Although there's a nested while, every index enters and leaves the stack once. Total push+pop ≤ 2n → linear overall, not quadratic.

See it: Daily Temperatures resolving on pop

Temps [73,74,75,71,76]. A warmer day pops all cooler days off the stack, and each pop records how many days they waited.

O(n). One warmer reading can resolve many waiting days at once — each index pops exactly once. ▶ loops every 8s.
incoming (warmer) resolved on pop on stack
⚠️ Common Pitfall

Decide upfront whether you need a strictly-increasing or strictly-decreasing stack, and push indices (not values) when you need distances. Using ≤ vs < in the pop condition changes how ties are handled — critical for histogram problems.

next greater/smallerhistogram rectanglepop on violationO(n) one passpush indices
📚 Pattern 13: Tree DP & Memoization

Dynamic programming on a tree: each node's answer depends only on its children's answers, computed bottom-up via post-order. The classic trick is returning a small tuple of states (e.g. "include this node" vs "exclude it").

🎯 When To Reach For It

  • Optimal decision at every tree node
  • House Robber III (rob/skip), tree diameter
  • Max path sum, independent set on trees
  • Subtree aggregates with constraints

🧩 The Core Idea

Recurse to children first (post-order). Combine their returned states into this node's answer. Return a tuple so the parent can choose based on whether this node was "used".

🔁 The Recurrence

def dfs(node) -> (rob, skip):
  l = dfs(left); r = dfs(right)
  rob = node.val + l.skip + r.skip
  skip = max(l) + max(r)
  return (rob, skip)

🌱 Base Case

A null child returns (0, 0) — nothing to rob, nothing to skip. The final answer is max(dfs(root)). One traversal, O(n).

See it: rob/skip values bubbling up

House Robber III. Each node returns (rob, skip). Leaves resolve first, then values combine upward — you can't rob a node and its child together.

O(n), single pass. Returning a (rob, skip) tuple lets the parent decide without recomputing children. ▶ loops every 9s.
leaf resolved subtree combined root answer
⚠️ Common Pitfall

Don't make two separate recursive calls for "rob" and "skip" — that's exponential. Return both states from one call. Forgetting the null base case (0,0) is the most common crash.

post-order DProb / skip tuplesubtree statesHouse Robber IIIO(n) one pass
📚 Pattern 14: Bitwise Operations & Bitmask DP

Treat an integer as an array of yes/no flags — one bit per element. This makes subsets cheap to enumerate and store as DP keys, and unlocks XOR tricks where pairs cancel out, leaving the odd one standing.

🎯 When To Reach For It

  • Subset enumeration / state compression
  • "Single number" XOR cancellation
  • Traveling salesman, assignment DP
  • n ≤ ~20 and you need all subsets

🧩 The Core Idea

Bit i set ⇒ element i is in the set. dp[mask] = best answer for exactly that subset. Transitions flip bits on/off. XOR is self-inverse: x ^ x = 0.

🔁 Handy Ops

x & (1<<i) # is bit i set?
x | (1<<i) # set bit i
x & (x-1) # clear lowest set bit
bin(x).count('1') # popcount

🌱 The XOR Insight

XOR every number in a list: pairs cancel to 0, so only the element appearing an odd number of times survives. No extra memory, one pass.

See it: XOR cancelling pairs

[4,1,2,1,2] — every number appears twice except 4. XORing them all left-to-right makes the duplicates annihilate, isolating 4.

O(n) time, O(1) space. No hash map needed — XOR's self-cancelling property does all the bookkeeping. ▶ loops every 8s.
the unpaired number pair cancels result
⚠️ Common Pitfall

Bitmask DP is only feasible for small n (≤ ~20–22) since there are 2ⁿ masks. Watch operator precedence: x & 1 << i parses as x & (1 << i) in some languages but not all — parenthesize to be safe.

bits = flagssubset = maskXOR cancellationdp[mask]n ≤ 20

5-Step Recognition Drill

How to classify any new problem fast

1. Look at the input shape. Array? Sorted? Linked list? Graph? Tree? Intervals? The structure narrows candidates immediately.
2. Read the ask. "Contiguous" → window. "All paths" → DFS. "Shortest" → BFS/binary search. "Top K" → heap. "Order with constraints" → topological.
3. Check the size limits. n ≤ 20 hints bitmask. n up to 10⁵ rules out O(n²) → think two-pointer/heap/binary search.
4. Name the state. What must you remember as you scan? That's your window variable, dp key, or stack invariant.
5. Pick the discard rule. Every efficient pattern throws work away — which half/region/branch can you safely skip?

Two Sum II (Input Array Is Sorted)
Problem (LeetCode 167): Given sorted array, find two numbers summing to target. Return 1-indexed positions. [2,7,11,15], target=9 → [1,2] (indices of 2 and 7)
Two Pointers: Initialize left=0 (start), right=n-1 (end). If sum equals target, return indices. If sum < target, move left forward (need bigger sum). If sum > target, move right backward (need smaller sum).
🔍 Step-by-Step Trace
arr=[2,7,11,15], target=9 left=0, right=3: sum=2+15=17 > 9 → right-- left=0, right=2: sum=2+11=13 > 9 → right-- left=0, right=1: sum=2+7=9 == 9 → FOUND ✓ Return [1, 2] (1-indexed)
💻 Solution
Python · Two Sum II
def two_sum(numbers, target):
    left, right = 0, len(numbers) - 1
    while left < right:
        s = numbers[left] + numbers[right]
        if s == target:
            return [left + 1, right + 1]
        elif s < target:
            left += 1
        else:
            right -= 1
⌛ Complexity
Time: O(n) Space: O(1)

def min_window(s, t):
    from collections import Counter
    required = Counter(t)
    formed = 0
    window_counts = {}
    l = 0
    best_len, best_l = float('inf'), 0
    
    for r in range(len(s)):
        window_counts[s[r]] = window_counts.get(s[r], 0) + 1
        if s[r] in required and window_counts[s[r]] == required[s[r]]:
            formed += 1
        
        while formed == len(required):
            if r - l + 1 < best_len:
                best_len = r - l + 1
                best_l = l
            lch = s[l]
            window_counts[lch] -= 1
            if lch in required and window_counts[lch] < required[lch]:
                formed -= 1
            l += 1
    
    return s[best_l:best_l+best_len] if best_len != float('inf') else ''

def shortest_path(grid, start, end):
    from collections import deque
    queue = deque([start + (0,)])  # (row, col, dist)
    visited = {start}
    dirs = [(0,1),(0,-1),(1,0),(-1,0)]

        r, c, dist = queue.popleft()
        if (r, c) == end:
            return dist

            nr, nc = r + dr, c + dc

                queue.append((nr, nc, dist + 1))
    
    return -1  # no path

def ladder_length(beginWord, endWord, wordList):
    from collections import deque
    word_set = set(wordList)
    if endWord not in word_set:
        return 0
    
    # Build pattern-to-words map
    pattern_map = {}
    for word in word_set | {beginWord}:
        for i in range(len(word)):
            pattern = word[:i] + '*' + word[i+1:]
            if pattern not in pattern_map:
                pattern_map[pattern] = []
            pattern_map[pattern].append(word)
    
    queue = deque([(beginWord, 1)])
    visited = {beginWord}
    
    while queue:
        word, dist = queue.popleft()
        if word == endWord:
            return dist
        
        for i in range(len(word)):
            pattern = word[:i] + '*' + word[i+1:]
            for neighbor in pattern_map.get(pattern, []):
                if neighbor not in visited:
                    visited.add(neighbor)
                    queue.append((neighbor, dist + 1))
    
    return 0

def can_finish(numCourses, prerequisites):
    from collections import deque, defaultdict
    
    indegree = [0] * numCourses
    graph = defaultdict(list)
    
    for course, prereq in prerequisites:
        indegree[course] += 1
        graph[prereq].append(course)

class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank = [0] * n
        self.components = n
    
    def find(self, x):
        if self.parent[x] != x:
            self.parent[x] = self.find(self.parent[x])
        return self.parent[x]
    
    def union(self, x, y):
        rx, ry = self.find(x), self.find(y)
        if rx == ry:
            return
        if self.rank[rx] < self.rank[ry]:
            rx, ry = ry, rx
        self.parent[ry] = rx
        if self.rank[rx] == self.rank[ry]:
            self.rank[rx] += 1
        self.components -= 1

def count_components(n, edges):
    uf = UnionFind(n)
    for u, v in edges:
        uf.union(u, v)
    return uf.components

def rob_tree(root):

def count_subset_sum(nums, target):
    dp = [0] * (target + 1)
    dp[0] = 1  # Empty subset sums to 0
    
    for num in nums:
        # Iterate backward to avoid using same element twice
        for s in range(target, num - 1, -1):
            dp[s] += dp[s - num]
    
    return dp[target]
