🧠 The Three Bounds — O, Ω, Θ
Why three notations?

Big O describes the worst case. Big Omega describes the best case. Big Theta describes both — a tight, two-sided bound.

When an interviewer asks "what's the complexity?", they almost always mean Big O (worst case). But understanding all three lets you give precise, impressive answers about algorithm behaviour under different inputs.

📚 Formal Definitions with Examples
Big O — upper bound (worst case)
f(n) = O(g(n)) iff ∃ constants c > 0, n₀ > 0 such that f(n) ≤ c·g(n) for all n ≥ n₀
Example: Linear search. Best case O(1) (first element). Worst case O(n) (last or not found). We say O(n).
Big Ω — lower bound (best case)
f(n) = Ω(g(n)) iff ∃ constants c > 0, n₀ > 0 such that f(n) ≥ c·g(n) for all n ≥ n₀
Example: Any comparison-based sort is Ω(n log n). No comparison sort can do better.
Big Θ — tight bound (both sides)
f(n) = Θ(g(n)) iff f(n) = O(g(n)) AND f(n) = Ω(g(n))
Example: Merge sort is Θ(n log n) — always n log n regardless of input.
📈 Visualising the Bounds

🔌 Best ≠ O, Worst ≠ Ω — Common Mistake

Big O is an upper bound — it does NOT mean "worst case". Big Omega is a lower bound — it does NOT mean "best case". They are mathematical bounds on a function.

🔧 Recurrence Relations — Master Theorem
Master Theorem: T(n) = a·T(n/b) + f(n) a = branching factor (sub-problems) b = division factor (how much smaller) f(n) = cost of work at each level Case 1: f(n) = O(n^(log_b(a) - ε)) → T(n) = Θ(n^log_b(a)) Case 2: f(n) = Θ(n^log_b(a)) → T(n) = Θ(n^log_b(a) · log n) Case 3: f(n) = Ω(n^(log_b(a) + ε)) → T(n) = Θ(f(n)) Common algorithms: Binary search: T(n)=T(n/2)+O(1) a=1,b=2, log_2(1)=0, f=O(1)=O(n⁰) → Case 2 → O(log n) Merge sort: T(n)=2T(n/2)+O(n) a=2,b=2, log_2(2)=1, f=O(n¹) → Case 2 → O(n log n) Strassen: T(n)=7T(n/2)+O(n²) a=7,b=2, log_2(7)≈2.81>2 → Case 1 → O(n^2.81)

1
Simplify T(n) = T(n-1) + O(n)
Question: Solve the recurrence T(n) = T(n-1) + O(n). This describes selection sort or any algorithm that does O(n) work at each level and reduces the problem by 1 each time.
Telescoping: Expand repeatedly until base case. T(n) = T(n-1)+n = T(n-2)+(n-1)+n = ... = n + (n-1) + ... + 1 = n(n+1)/2 = O(n²). This is the sum of arithmetic series.
🔍 Step-by-Step Trace
T(n) = T(n-1) + n Expand (telescoping): T(n) = T(n-1) + n T(n-1) = T(n-2) + (n-1) T(n-2) = T(n-3) + (n-2) ... T(2) = T(1) + 2 T(1) = O(1) (base) Substitute back: T(n) = O(1) + 2 + 3 + ... + (n-1) + n = 1 + 2 + 3 + ... + n = n(n+1)/2 = O(n²) Algorithms with this recurrence: - Selection sort (find min in shrinking array) - Insertion sort (worst case: already reverse sorted) - Naive string matching
💻 Solution
Python · Verify empirically
def count_ops(n):
    # T(n) = T(n-1) + n pattern: e.g. selection sort comparisons
    total = 0
    for i in range(n):
        for j in range(i+1, n):   # inner loop shrinks
            total += 1
    return total

for n in [10, 100, 1000]:
    ops = count_ops(n)
    formula = n*(n-1)//2
    print(f"n={n}: ops={ops}, n(n-1)/2={formula}")
# n=10: ops=45, n(n-1)/2=45 ✓
⌛ Complexity
Time: O(n²) Space: O(1)
2
Compare Θ vs O for Merge Sort vs Quick Sort
Question: Merge sort is O(n log n). Quick sort is also O(n log n). But merge sort is Θ(n log n) while quick sort is NOT Θ(n log n). Explain why.
Θ requires both bounds to hold for ALL inputs. Merge sort always divides evenly, always does O(n) merge work — the recurrence is always T(n)=2T(n/2)+O(n) regardless of input → Θ(n log n). Quick sort's recurrence depends on pivot: with sorted input and bad pivot, T(n)=T(n-1)+O(n) → O(n²). So O(n²) worst case means it cannot be Θ(n log n).
🔍 Step-by-Step Trace
Merge sort on any input: Level 0: 1 problem of size n → n work Level 1: 2 problems of n/2 → n work Level 2: 4 problems of n/4 → n work ... log n levels, n work each = Θ(n log n) always Quick sort — good pivot (median): Level 0: 1 problem of size n → n work Level 1: 2 problems of n/2 → n work (balanced!) → Θ(n log n) this time Quick sort — bad pivot (always smallest): Level 0: 1 problem of size n → n comparisons Level 1: 1 problem of size n-1→ n-1 comparisons ... n levels total → Θ(n²) this time Since worst case is Θ(n²), we cannot say T(n)=Θ(n log n). We CAN say T(n)=O(n²) and T(n)=Ω(n log n).
💻 Solution
Python · QuickSort worst case demo
import sys; sys.setrecursionlimit(10000)
def qsort_worst(arr):
    # Always pick first as pivot → worst on sorted input
    if len(arr) <= 1: return arr
    pivot = arr[0]
    left  = [x for x in arr[1:] if x <= pivot]
    right = [x for x in arr[1:] if x  > pivot]
    return qsort_worst(left) + [pivot] + qsort_worst(right)
# sorted([1..1000]) triggers O(n²) — 1000 levels of recursion!
⌛ Complexity
Time: O(n²) worst, O(n log n) avg Space: O(n) stack

What is o(n) (little o) vs O(n)?
Question: What is the difference between o(n) and O(n)? Give an example function that is O(n) but not o(n).
Little-o is strictly less than; Big-O is at most. f(n) = o(g(n)) means f grows STRICTLY SLOWER than g — for every c > 0, there exists n₀ such that f(n) < c·g(n). Big-O allows equality. So O(n) includes Θ(n), but o(n) excludes it.
🔍 Step-by-Step Trace
o(n) vs O(n): f(n) = 5 is o(n)? YES: 5 < c·n for all n>5/c (constant is strictly slower than linear) f(n) = log n is o(n)? YES: log n grows strictly slower than n f(n) = n is o(n)? NO: n ≥ c·n means c≤1, but "for EVERY c" must hold; for c=1, n<1·n is false f(n) = n is O(n)? YES: n ≤ 1·n for all n ≥ 1 ✓ f(n) = n is o(n)? NO: for c=1, we'd need n<1·n=n which is false ✗ So linear search T(n) = n is O(n) but NOT o(n). It's Θ(n) — tight bound. Analogy: O(n) like ≤ (at most) o(n) like < (strictly less than) Ω(n) like ≥ (at least) ω(n) like > (strictly greater than) Θ(n) like = (exactly)
